## Critical Findings

### C1. `install.sh` `DEFAULT_VERSION="0.1.0"` is five releases behind HEAD
- **File:** `install.sh:16`
- **What:** Default `VEDA_VERSION` is `0.1.0`. Current release is `0.1.5`. Anyone curling the script without setting `VEDA_VERSION` gets an outdated CLI (no recursive cp, no grep, no `--version`, no `score_type`, no real BM25 client behavior, no friendly fuse-missing message). The very script that's supposed to give friends a one-shot install hands them a year-old binary.
- **Why it matters:** The whole point of v0.1.5 + the `/skill` install path is "have a friend run this and try the new flow." A stale default silently undoes that.
- **Recommendation:** Either (a) bump `DEFAULT_VERSION` in `scripts/release.sh` as part of the tag-bump step (one extra `sed -i`), or (b) replace the default with a `latest`-tag fetch — `curl -fL https://api.github.com/repos/$GITHUB_REPO/releases/latest | jq -r .tag_name` — and only fall back to a hardcoded version if the API call fails. (a) is simpler and matches the existing release.sh contract.

### C2. `install.sh` post-install instructions print an invalid command
- **File:** `install.sh:164`
- **What:** Tells the new user to run `veda workspace create my-workspace && veda use my-workspace`. There is no `veda use` subcommand — `Commands` only has `Workspace { action: WorkspaceCmd }`, and the use subcommand is `veda workspace use <id>`. Worse, `<id>` is a UUID returned by `workspace create`, not the workspace's name. Both halves of `&&` will fail: the right side because the subcommand doesn't exist; if it did, the workspace name string wouldn't resolve to a workspace_id. (`crates/veda-cli/src/main.rs:154-168` defines the real `WorkspaceCmd::Use { id }`.)
- **Why it matters:** First-run onboarding step #4 visibly fails. The user has just curled a script over the network — confidence is exactly the thing breaking here matters most.
- **Recommendation:** Print the real flow:
  ```
  veda workspace create my-workspace      # prints the workspace id
  veda workspace use <id-from-above>      # selects + creates a workspace key
  ```
  Or, since most users won't manually copy the id, add a `--switch` flag to `workspace create` that does the use step inline. That's a real UX gap, not just a docs fix.

### C3. `skill.md` claims PDF text auto-extraction; CLI rejects non-UTF-8
- **File:** `skill.md:65`
- **What:** The skill tells AI agents `veda cp ./report.pdf /docs/report.pdf  # PDF — text auto-extracted`. But (1) `crates/veda-cli/src/main.rs:294` uses `std::fs::read_to_string`, which returns `InvalidData` for any non-UTF-8 input — a binary PDF won't even leave the client; and (2) `ARCHITECTURE.md:31` confirms `extract_text` is still a `text/plain` placeholder, no PDF backend wired up. This is also self-contradicted on `skill.md:215` ("PDFs/images keep extracted text only").
- **Why it matters:** Agent will confidently `veda cp ./foo.pdf` based on the skill, hit `read_to_string` failure, and report something nonsensical. Worse, an LLM may infer wrongly that the PDF is uploadable but binary-stripped — a believable-but-wrong story.
- **Recommendation:** Drop the PDF example. The "Don't" section already tells agents PDFs aren't a thing — keep that, remove the misleading positive example.

### C4. `skill.md` warns about a leak that no longer exists
- **File:** `skill.md:158-162, 221-222`
- **What:** Tells agents to "always strip the `vector` field" from `veda collection search` output before feeding to an LLM. But the v0.1.5 D1 fix (`crates/veda-core/src/service/collection.rs:184-194`) already strips `vector` and `workspace_id` server-side. The advice is now wrong about the leak path and creates uncertainty for agents about what other fields they should defensively strip.
- **Why it matters:** Stale defensive guidance is worse than no guidance — the agent spends tokens on a non-issue and may grep for fields that aren't there. Also undermines trust in the rest of the skill (if THIS warning is wrong, what else?).
- **Recommendation:** Replace the warning with a one-liner like "v0.1.5+ server strips internal fields (`vector`, `workspace_id`); show results as-is." Keep the suggestion to prefer `veda sql` for tabular output for unrelated reasons (cleaner format).

### C5. Destructive auto-migration has no progress signal and a stuck-state failure mode
- **File:** `crates/veda-store/src/milvus.rs:812-841`, `crates/veda-server/src/main.rs:99-107, 315-380`
- **What:** On startup, if the existing `veda_chunks` collection lacks `sparse_vector`, the code drops + recreates it (every chunk in every workspace, gone), then walks every dentry of every workspace and enqueues a `force_reembed=true` ChunkSync for each. The intent is sound (Milvus REST has no `ALTER COLLECTION ADD FIELD`), but:
  - No metric / log progress for the enqueue walk. A 100k-file workspace is silent for many seconds during startup before the server listens.
  - `has_pending_event` is called per-file in the inner loop. The 2026-04-30 review (§3.7) flagged that `has_pending_event` does a `JSON_EXTRACT` table scan; here that's O(N²) on workspace size. For a 50k-file workspace this can dominate startup — and the migration runs **synchronously before the listener binds**, so the server is unreachable the entire time.
  - If the process is killed mid-walk, on next boot `collection_has_sparse_vector` returns true (the new schema already exists) so the migration won't re-trigger. The system is in a "0.1.4 worth of files, 0.1.4-N worth of chunks in Milvus" stuck state until the reconciler's 6-hour pass picks it up. That recovery path works, but it's not documented and is invisible to the user.
- **Why it matters:** First-time upgrade from v0.1.4 → v0.1.5 in a populated workspace can mean: (a) startup hangs for tens of seconds, (b) if interrupted, search returns zero hits for up to 6h, with no surfaced indicator. Joe specifically sized this for a single-tenant alpha, so the blast radius is small — but the next time the same destructive-on-schema-mismatch pattern is reused (e.g. another field addition), the same trap re-triggers.
- **Recommendation:**
  1. Add a counter: `metrics::counter!("veda_migration_chunksync_enqueued").increment(N)` per page. Lets ops grep `/v1/metrics` to see the walk completed.
  2. Add a "migration_in_progress" gauge that the worker / reconciler can read so they can fast-path through stuck-state detection.
  3. Document in `docs/design/` the migration contract: schema-shape mismatch ⇒ destructive rebuild ⇒ reconciler is the safety net. Right now this is encoded only in the code comment chain.
  4. Consider running the reembed walk **after** the listener binds (in a `tokio::spawn`), guarded by a `migrating=true` flag that `/v1/ready` reflects. Server is reachable for liveness during the walk; clients see `503` from `/v1/ready` and back off.

## Strategic Concerns

### S1. `summary_sync` still loads the entire chunked file into memory
- **File:** `crates/veda-server/src/worker.rs:423-433`
- **What:** B2 (`handle_chunk_sync`) was rewritten to paginate chunks in batches of 32 (`worker.rs:248-275`), capping peak memory at one batch + the assembled `String`. But `handle_summary_sync` still does `self.meta.get_file_chunks(file_id, None, None).await?` and calls `.collect::<String>()` — the very pattern B2 was added to remove. For a 50MB chunked file this allocates ~100MB peak (Vec<FileChunk> + assembled String) per concurrent SummarySync task in the batch, and the batch_size default is 10.
- **Why it matters:** One bad burst of summaries on large files plus the existing 50MB cap and concurrent batch can briefly use ~1GB RSS — fine on dev, painful on a 2GB virtual machine alpha.
- **Recommendation:** Reuse the same paginated path B2 added (`list_chunk_byte_lens` + `get_chunks_in_index_range`). The two handlers should share a helper `load_full_content(meta, file)` that does the right thing for both inline and chunked storage.

### S2. `grep` service has no time bound and reads each candidate file fully
- **File:** `crates/veda-core/src/service/fs.rs:683-742`
- **What:** Walks up to 50,000 dentries; for each non-dir, calls `read_file` (loads the whole file into memory) and scans line-by-line. No deadline, no cumulative-time check, only an early-exit at `max_results`. A pathological prefix (workspace with 50k chunked files near the 50MB cap, none matching) will pull ~2.5TB through the storage layer before returning.
- **Why it matters:** alpha `veda grep` is positioned in `skill.md` as the "fast literal substring scan" choice over `search --mode fulltext`. The reality is that for non-matching queries it's a full-workspace I/O storm. The HTTP request timeout (60s client-side) saves the user but the server keeps grinding for the rest of the request lifetime.
- **Recommendation:** Add an early-exit on a server-side wall-clock budget (e.g. 30s). Also consider streaming line scan via the chunked-paginated reader rather than loading whole files — the structure is already there from B2. Not blocking for alpha if the cap stays at 50k entries, but document the cost in `alpha-tryout.md`.

### S3. Multilingual detector labels Japanese/Korean as `zh-CN`
- **File:** `crates/veda-pipeline/src/summary.rs:111-132, 322-330`
- **What:** Hiragana, Katakana, and Hangul codepoints are counted as CJK; `>25% CJK` returns the literal string `"zh-CN"`. Tests *encode* this behavior (`detect_lang_hiragana_katakana_hangul_count_as_cjk`) so it's intentional. The prompt then injects "Output language: zh-CN" — a Japanese README will get a Chinese summary, which is wrong.
- **Why it matters:** Soft fail — content is comprehensible, just wrong language. For alpha targeted at zh+en users, the cost is low. But the test enshrines the bug, so future fixes will look like regressions.
- **Recommendation:** Either return separate `"ja"` / `"ko"` labels (LLM handles them fine), or make the slot literal `"detected as the source's primary language"` and let the LLM pick. The current approach is the worst of both — explicit-but-wrong.

### S4. `permissions: contents: write  # needed by softprops/action-gh-release` — comment outdated
- **File:** `.github/workflows/release.yml:14`
- **What:** The comment names a tool that the workflow no longer uses (commit `43f5d4e` switched to `gh release create`). Permission is still needed, so the line is correct — but the comment is misleading the next person who edits this file.
- **Why it matters:** Trivially small; flagging because comment-rot like this compounds.
- **Recommendation:** Change to `# needed by 'gh release create'`.

### S5. No env override for `VEDA_EMBEDDING_BATCH_SIZE`
- **File:** `crates/veda-server/src/config.rs:213-334`
- **What:** Every other embedding/MySQL/Milvus/llm/retention/reconciler/dev_mode/metrics field has a `VEDA_*` env override. `embedding.batch_size` (added in commit `346a923`) does not. Operationally minor, but inconsistent — the config-only field stands out.
- **Recommendation:** Add `VEDA_EMBEDDING_BATCH_SIZE` to `apply_env_overrides`. Three lines.

### S6. `init_collections` rebuilt-bool plumbing is correct but hidden
- **File:** `crates/veda-store/src/milvus.rs:812`, `crates/veda-server/src/main.rs:99-107`
- **What:** `init_collections` returns `bool` to tell the caller "I just dropped your data — please re-enqueue it." This works as-is, but the contract is implicit in a comment and a single boolean. If a future schema change adds a *second* destructive migration path (e.g. summary collection), the boolean conflates them and the caller-side `reembed_all_files_for_migration` would have to know to also do `reembed_all_summaries`.
- **Why it matters:** Architecturally fine for v0.1.5; will get awkward if there's a v0.1.7 that does the same dance for the summary collection.
- **Recommendation:** Soft suggestion: when the next destructive migration ships, replace the bool with a `Vec<RebuiltCollection>` enum and dispatch on each. Don't refactor preemptively.

## What's Good

Genuinely worth preserving — these decisions show real care:

1. **`milvus.rs` `post()` vs `post_no_retry()` split** (lines 99-127). The comment `Use ONLY for idempotent endpoints… Non-idempotent mutations (collections/create, entities/insert) MUST go through post_no_retry` is exactly the kind of safety contract that gets undocumented in normal projects and then violated 18 months later. Encoding it in the API name + comment is a long-life decision.

2. **`SearchHit::score_type` doc comment** (`crates/veda-types/src/types.rs:331-334`). "Scores are NOT comparable across types — agents/clients must read this field before reasoning about magnitude" is the kind of foot-gun callout that pays for itself once a downstream agent fuses scores from `bm25` (~[0,30]) and `cosine` (~[0,1]).

3. **`get_summary` 501 vs 202 distinction with `Cache-Control: no-store`** (`routes/search.rs:64-89`). RFC 7231 lets clients cache 501 by default; explicitly setting no-store so a config flip doesn't get pinned by proxies is exactly right. The CLI exit-code differentiation (2 vs 3) is the icing.

4. **Worker `catch_unwind` panic isolation** (`worker.rs:87-104`). A panic in one task no longer kills the entire worker future and leaves leases stuck. Plus the three unit tests for `panic_message` extraction (static str / owned String / non-string payload).

5. **Paginated chunk reading in `handle_chunk_sync`** (`worker.rs:248-275`). The comment "caps peak memory at ~file_size + 1 batch (was ~2x file_size when we held the full Vec<FileChunk> AND the assembled buf)" is honest about the prior cost and why it was worth fixing.

6. **`init_collections` rebuilt-bool → main.rs paginated re-enqueue with `force_reembed`**. The whole signal chain works; even with the C5 caveats above, the *concept* is the right architecture for "Milvus REST can't ALTER, so destroy-then-rebuild and let the reconciler net catch any drops." The paginated walk via `list_dentries_under_page` is exactly the C2 fix from the prior review applied here too.

7. **`cp_dir_recursive` symlink skip** (`main.rs:608-612`). Avoids both infinite-recursion and the symlink-escape attack ("a symlink in source tree pointing at /etc could exfiltrate"). Comment explains the reasoning, not just the mechanism.

8. **Summary debounce + burst window with explicit honesty** (`worker.rs:359-380`). The comment "the only leak is the first edit of a brand-new file — accepted" is the kind of trade-off acknowledgment that survives scrutiny months later. Plus the metric `veda_summary_enqueue_total{burst="in_burst"|"isolated"}` makes the trade-off measurable.

9. **`metrics_token_empty_env_stays_disabled` test + comment** (`config.rs:464-473`). "Empty env must not silently enable an empty-token bypass" — exactly the "explicit-disable" muscle memory needed for security knobs.

10. **`hybrid_search_remote` comment about RRF degeneracy** (`milvus.rs:504-508`). "Sending only the dense object (as the previous code did) made RRF a no-op identity reorder — hybrid degenerated into semantic." Captures exactly why the previous code was subtly wrong without being defensive about it.

### Key Decisions Observed

- **MySQL = control plane, Milvus = data plane** — confirmed; v0.1.5 doesn't break this. BM25 lives in Milvus, not MySQL.
- **Destructive auto-migration on Milvus schema mismatch** (chunks collection drop + paginated re-enqueue ChunkSync) — new contract introduced in v0.1.5. Reconciler is the documented-in-comments safety net. Should be lifted into a design doc.
- **Score-type plurality at the type level** (`SearchHit::score_type: String`) — accepts that hybrid/bm25/cosine fuse into one `score: f32` slot but tags the meaning. Better than two separate fields.
- **Async-summary tri-state** (200 ready / 202 pending / 501 disabled) — encoded both at server (status + Cache-Control + Retry-After) and client (different exit codes). Worth keeping as a pattern for the next "feature exists but is gated on config" surface.
- **Worker resilience over throughput** — `catch_unwind` per task, paginated chunk reads, debounced summary enqueue with burst detection. The trade-off everywhere is "lose a little speed/cost, never the worker process or memory headroom." Fits an alpha-period dogfooding stance.
- **Public install vs. private network** — install.sh hardcodes a private 10.79.51.161 default server. This is a deliberate alpha choice (Joe + friends on the same network). Once the audience widens, it'll need to become an empty default that prompts.
