## Verdict

当前实现**不建议直接发布**。核心能力已经连起来了（Auth/FS/Search/Collection/SQL/CLI），但在“API 契约一致性、目录语义正确性、Collection SQL 正确性、Worker 幂等性”上存在高风险缺陷，部分会导致功能直接不可用或数据一致性受损。  
建议至少修复 Critical/High 问题并补充针对性测试后再合并。

## Critical Findings

### 1) CLI `mv` 与服务端契约不一致，命令实际不可用
- **File:** `crates/veda-cli/src/client.rs:101`、`crates/veda-server/src/routes/fs.rs:120`
- **What:** CLI 发送 `{"src","dst"}`，服务端解析 `{"from","to"}`。
- **Why it matters:** `veda mv` 会稳定失败（参数解析失败/400），属于直接功能故障。
- **Recommendation:** 统一字段名（建议都改为 `from/to` 或都改为 `src/dst`），并加 CLI-to-server 集成测试覆盖 `mv`。

### 2) SQL Collection 查询使用了错误的物理集合名，线上会查不到数据
- **File:** `crates/veda-sql/src/collection_table.rs:83`、`crates/veda-core/src/service/collection.rs:108`
- **What:** SQL 层查询使用 `schema.name`，但写入/搜索 collection 时真实 Milvus 名是 `veda_coll_{collection_id}`。
- **Why it matters:** DataFusion 中 `SELECT * FROM <collection_name>` 在真实后端会命中错误 collection，结果为空或报错。
- **Recommendation:** 在 SQL 层统一使用 `schema.id -> veda_coll_{id}` 的映射，禁止直接用逻辑名访问 Milvus 物理表。

### 3) 目录重命名仅更新单个 dentry，子树不会迁移
- **File:** `crates/veda-core/src/service/fs.rs:499`、`crates/veda-store/src/mysql.rs:771`
- **What:** `rename` 只更新当前路径，未递归更新子项的 `path/parent_path`。
- **Why it matters:** 目录 rename 后，子文件路径与目录结构不一致，导致“目录树破坏/不可预期访问”。
- **Recommendation:** 对目录 rename 实现子树批量重写（事务内），并补充目录级 rename 回归测试。

### 4) Worker 忽略删除失败 + 非幂等 chunk id，可能产生陈旧/重复向量
- **File:** `crates/veda-server/src/worker.rs:129`、`crates/veda-server/src/worker.rs:138`
- **What:** chunk id 每次随机；删除旧向量失败被 `.ok()` 吞掉。
- **Why it matters:** 删除失败时会继续写入新 chunk，导致重复/脏数据累积，搜索质量下降且难排查。
- **Recommendation:**  
  1) 将 chunk id 改为稳定键（如 `file_id:chunk_index`）；  
  2) 删除失败应直接 fail 当前任务，走重试；  
  3) 加监控指标（delete_fail/upsert_fail）。

## High Findings

### 5) `read_file_lines` 在 `start > end` 时可能 panic
- **File:** `crates/veda-core/src/service/fs.rs:219`
- **What:** 切片 `lines[s..e]` 未保证 `s <= e`。
- **Why it matters:** 恶意或错误请求可能触发 handler panic，造成可用性风险。
- **Recommendation:** 在入口校验 `start >= 1 && end >= start`，不满足返回 `InvalidInput`。

### 6) `copy_file(src == dst)` 会错误增加引用计数
- **File:** `crates/veda-core/src/service/fs.rs:409`
- **What:** 先 `increment_ref_count`，且无 `src == dst` 的短路；目标已存在且同 file_id 时不会回滚增量。
- **Why it matters:** ref_count 漂移会导致删除回收逻辑异常（泄漏或延迟回收）。
- **Recommendation:** 在 `copy_file` 最前面加 `if src == dst { return Ok(current_info) }`。

### 7) Search 结果 path 为空且 `path_prefix` 未生效
- **File:** `crates/veda-store/src/milvus.rs:238`、`crates/veda-core/src/service/search.rs:32`
- **What:** Milvus 命中转 `SearchHit` 时固定 `path: None`；`path_prefix` 参数未在向量后端实现。
- **Why it matters:** 返回结果缺少可定位路径，且无法按目录过滤，不符合 API 语义。
- **Recommendation:** 搜索结果回填 `file_id -> path`（可批量查 metadata），并实现 path 前缀过滤。

### 8) MySQL 路径唯一索引使用前缀 255，长路径存在冲突风险
- **File:** `crates/veda-store/src/mysql.rs:342`
- **What:** `UNIQUE INDEX idx_ws_path (workspace_id, path(255))` 只保证前缀唯一。
- **Why it matters:** 长路径（>255）同前缀时会误判冲突，破坏文件系统语义。
- **Recommendation:** 按设计改为 `path_hash` 唯一索引，或使用完整可索引哈希列。

## Strategic Concerns

### 1) SQL 表实现语义偏离“关系查询”，会导致不完整结果
- **File:** `crates/veda-sql/src/collection_table.rs:81`
- **What:** 用 ANN search + `limit=16384` 拉取“全表”数据。
- **Why it matters:** 这不是全量 scan，数据规模增长后 SQL 查询会 silently truncation。
- **Recommendation:** 增加“按过滤条件 query/scan”路径，不要用向量检索模拟全表读取。

### 2) API 契约出现多处漂移（设计文档 vs 类型定义 vs 实现）
- **File:** `docs/design.md:604`、`crates/veda-types/src/api.rs:119`、`crates/veda-server/src/routes/sql.rs:14`
- **What:**  
  - 文档约定 `fs` copy/rename query 风格，代码为独立 endpoint。  
  - `SqlRequest` 类型字段为 `query`，服务端实际接收 `sql`。  
  - SQL 响应结构与 `SqlResponse` 类型未对齐。
- **Why it matters:** SDK/CLI/第三方接入会出现“类型正确但调用失败”。
- **Recommendation:** 以单一来源统一 API 契约（建议以 `veda-types` + OpenAPI 生成为准），并加契约测试。

### 3) Collection 元数据与 Milvus 操作缺少补偿策略
- **File:** `crates/veda-core/src/service/collection.rs:65`
- **What:** `create/delete` 跨 MySQL 与 Milvus 的步骤非事务，无失败补偿。
- **Why it matters:** 部分失败会留下孤儿 collection 或脏 schema。
- **Recommendation:** 增加补偿（失败时回滚另一侧）或引入 outbox 驱动的最终一致流程。

### 4) 测试不能有效防回归（尤其 server 层）
- **File:** `crates/veda-server/tests/server_test.rs:167`
- **What:** 集成测试自建了“最小 server”，未复用真实 `routes::build_router`。
- **Why it matters:** 真实路由/鉴权/序列化问题（如 `mv` 参数不一致）不会被该测试捕获。
- **Recommendation:** 用真实 app state + 真实 router 做黑盒测试；把关键流（account→workspace→fs/mv/search/sql）纳入必跑集。

## Redundancy & Waste

- `crates/veda-server/tests/server_test.rs` 与正式路由逻辑重复实现，维护成本高且易漂移。  
- `crates/veda-cli/src/client.rs` 发送 `embedding_source`，但服务端 `CreateCollectionRequest` 不接收，属于无效输入。  
- `crates/veda-sql/src/collection_table.rs` 将 int/float 统一映射为 Utf8，丢失类型信息，后续补齐成本高。

## Documentation Gaps

- `docs/design.md` 中 FS copy/rename API 与当前实现不一致。  
- SQL 请求/响应字段定义与 `veda-types`/`veda-server` 存在冲突，缺少统一说明。  
- 文档未明确 Collection 逻辑名与 Milvus 物理名的映射规则（`name` vs `veda_coll_{id}`），导致 SQL 层实现偏差。

## What's Good

- crate 分层总体清晰，`veda-core` 与存储实现解耦保持得不错。  
- Auth 模型（Account/Workspace + API key/workspace key/JWT）实现结构与设计方向一致。  
- MySQL/Milvus 集成测试框架已具备，后续补全关键场景成本不高。

### Key Decisions Observed

- 当前实现将 Collection 的“逻辑名”和“Milvus 物理名”分离，但分离规则未在跨模块统一，导致 SQL 层走偏。  
- FS 以 `dentry + file + outbox` 建模，方向合理，但目录 rename/copy 边界尚未封实。  
- Worker 采用“先删后写”的更新模型，但缺少失败硬约束与幂等键，放大了异常路径风险。
