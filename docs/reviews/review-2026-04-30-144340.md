## 总体评价（Verdict）

这是一个**架构设计扎实、注释质量极高、防御性思考成熟**的 alpha 阶段项目。能看出作者在并发、一致性、failure mode 上花了非常多功夫——很多注释把"为什么这样做"讲得清清楚楚，水平远高于业余项目。但**距离生产可用还有明显距离**：未做账户级权限的悬挂位（suspended account 仍可用 key）、未做速率限制和资源上限、Reconciler 在大 workspace 下的 N+1 与全表扫描、FUSE 截断的 read-modify-write、CORS 默认 permissive 等问题，会在第一个真实租户上暴露。

**判断**：
- 给朋友试用（README 里描述的 alpha 场景）：**可以**。
- 给一个 5-50 人的团队当生产工具（design.md 设定的目标）：**不行**，需要把"批判性发现"清单中的至少 #1–#7 全部解决再考虑。
- 单租户自用：**完全可以**，且实现质量高于市面上大多数同类型自托管工具。

## 1. 架构层面（最重要）

### 1.1 优点（保留这些决策）

| 决策 | 评价 |
|---|---|
| **8-crate 拆分** | 边界清晰：`veda-types` 零依赖、`veda-core` trait + 业务、`veda-store` 存储实现、`veda-sql` 隔离 DataFusion 重编译。CLI/FUSE 不拉存储层。这是教科书级的分层。 |
| **MySQL = 控制面，Milvus = 数据面** | Outbox + 异步 Reconciler 自愈是正确的弱一致性方案。比"分布式事务"和"双写"都健壮。 |
| **Outbox 模式 + `FOR UPDATE SKIP LOCKED`** | 多 worker 安全，崩溃后 `lease_until` 过期可被其他 worker 接管。retry_count 在重新认领时只递增一次（Codex 已修过）。 |
| **AppKey/WorkspaceKey/JWT 三层凭证** | 仿 db9.ai 的设计，分管控面 / 数据面 / 短期会话，比单一 token 灵活。 |
| **`upsert_chunks_only` + 末尾的一次 `delete_chunks_above`** | 将 Milvus 部分批失败的"末尾空洞"窗口关掉。这是教科书级的 batched-写隐患处理（worker.rs:259-267 的注释把 trade-off 写得非常清楚）。 |
| **`last_embedded_content_hash` 水印 + 显式 `force_reembed` 标志** | 把"已 embed 过"和"Milvus 实际有数据"解耦。Reconciler 走 force_reembed 路径绕过水印——这是真懂 reconciler 模式的人才会写的细节。 |
| **SSE 410 协议带 `current_min_id` + `current_max_id`** | 客户端可以按 `max_id` 重订阅 → 保证 list_dir 重同步窗口里不会漏掉新事件，把竞态缝合住了。 |
| **`compute_write_meta` 走 `spawn_blocking`** | SHA256 + chunk 切分这种 CPU 重活不会饿死 tokio worker 线程。这一类异步 hygiene 在整个项目里非常一致。 |

### 1.2 战略性结构问题

#### S1（重要）：缺少"workspace 容量上限"的统一 enforcement 点
- `MAX_FILE_BYTES = 50MB`、`MAX_LINE_RANGE = 100k` 是**单文件**上限。
- 没有"workspace 总文件数 / 总字节"上限。`StorageStats` 只是查询，不做拒绝。
- `list_dentries_under(workspace_id, "/")` 在 Reconciler 和 FilesTable 里**无分页地读全表**——一个有 100 万文件的 workspace 直接 OOM 服务器（见 #C2）。
- 如果将来要 SaaS 化，**必须在 Account/Workspace 层加 quota 字段并在 write_file/append_file/insert_dentry 入口校验**。当前仅靠 `MAX_FILE_BYTES` 不能阻止恶意租户灌爆磁盘。

#### S2（重要）：`AccountStatus::Suspended` 是死字段
- `auth.rs::AuthAccount::from_request_parts` 走 `get_api_key_by_hash`，SQL 是 `WHERE key_hash = ? AND status = 'active'`——**只查 key 状态，不查 account 状态**。
- 同理 `get_workspace_key_by_hash` 不 join account 表。
- `WorkspaceKey` 路径在 auth.rs:178 检查了 `WorkspaceStatus::Active`，但同样没有检查 `Account.status`。
- **结果**：把 account 标记为 suspended 之后，已经下发的 API key / Workspace key / JWT 全部仍可用，相当于"封号"功能不存在。
- 修复成本极小：在 key 查询里 join `veda_accounts.status` 或在中间件里追加一次 `get_account` 校验（命中率高的 API 请预热缓存）。

#### S3（中）：`CollectionService::create` 的 saga 不闭环
```rust
// veda-core/src/service/collection.rs:71-85
self.vector.create_dynamic_collection(...).await?;       // 1. Milvus 建表
if let Err(e) = self.meta.create_collection_schema(&cs).await {  // 2. MySQL 写 schema
    if let Err(drop_err) = self.vector.drop_dynamic_collection(...).await {  // 3. Milvus 回滚
        error!(...);  // 仅记录日志，不重试
    }
    return Err(e);
}
```
- 步骤 3 失败 → Milvus 留下孤儿 collection，永远不被回收。
- 进程在步骤 1 后崩溃 → 同样的孤儿。
- Reconciler 仅对接 `veda_chunks` 和 `veda_summaries`，**不管 collection schema 漂移**。
- 修复路径：把 Milvus 创建延迟到 schema 写入成功后；或者在 Reconciler 里加一遍 collection schema 的 MySQL ↔ Milvus diff。

#### S4（中）：写路径串行编排，缺失并发 worker 协调
- 所有写都进同一个 `veda_outbox` 表，单 worker 进程从该表 `claim`。多副本部署时多 worker 抢同一表（`SKIP LOCKED` 是对的）。
- 但 `Worker::poll_once` 用 `for_each_concurrent(batch_size)` 跑同一批的所有任务，**没有按 file_id 串行化**。极端情况：一个文件被快速写两次 → 两个 ChunkSync 同时 claim（都在不同批被领走）→ 两个 worker 同时 embed 同一个 file_id 的 chunks → Milvus 上同一 `id = file_id_chunk_index` 的 upsert 互相覆盖（因为 upsert 是按 id 幂等的，结果其实没问题）。
- 但 `update_file_content_hash(file_id, hash)` 写 last_embedded_content_hash 时是 **last-writer-wins**——后跑完的写入若用的是 stale `file.checksum_sha256`，会把水印写成旧 hash，下一次相同内容来 ChunkSync 时 short-circuit 失败、白干一次 embed。这种"幂等性 OK 但成本浪费"的微妙竞态是个长期隐患。
- 修复：worker 在 process_task 之前对 file_id 做 in-process mutex（per-process），或者在 `try_insert_outbox_for_file` 已经做的 dedup 基础上让"已 processing 的文件"不进入新 batch。

#### S5（中）：Milvus REST v2 的非幂等写无去重 ID
- `upsert` 操作通过 primary key id (`file_id_chunk_index`) 是幂等的——OK。
- `insert_collection_rows` 走 `entities/insert`（**非幂等**），且没有客户端生成稳定 id 的强制路径。
  - 实际看 `CollectionService::insert_rows` 在 entry 时生成了 `obj.entry("id").or_insert_with(|| Uuid::new_v4())`——**只有 row 没自带 id 才填 UUID**，调用方传同样的 row 两次会得到不同 id，造成重复行。
  - 设计上 OK（用户可以传幂等 key），但默认行为是"重试不安全"，应该在 OpenAPI/CLI 文档明确。

#### S6（中）：DataFusion SQL 把整个 workspace 暴露成可查询表
- `VedaSqlEngine::execute` 注册 `files`、所有 collection、`veda_fs()` UDTF、`veda_fs_events()`、`search()` 等。
- 一个 read-only workspace key 持有人可以执行 `SELECT veda_read(path) FROM files`——把整个 workspace 文件内容通过 UDF 串行拉出来。这本身不是漏洞（对方有合法的读权限），但：
  1. 资源消耗放大器：单条 SQL 触发 N 次 `read_file`、N 次 stat、N 次 embed (`embedding()` UDF)。
  2. 没有查询超时/资源限制。一条 `SELECT search('xxx', 'hybrid', 100) FROM files` 触发 N 次 vector search。
- 建议：给 SQL 加全局超时 + per-row UDF 调用计数；或文档明确 SQL endpoint 不适合给非可信调用方。

#### S7（小但易踩）：`crates/veda-fuse` 不在 workspace members 中
```toml
exclude = ["crates/veda-fuse"]
```
- 后果：`cargo test`（CI 默认）**不会跑 FUSE 任何代码** —— 编译错误、单测失败都不会暴露。
- 当前 release log 里 FUSE 改动占很大比重，这是个明确的工程风险。
- 修复：在 CI 里加一步 `cargo test --manifest-path crates/veda-fuse/Cargo.toml`（macOS+Linux 矩阵），或者条件编译进 default workspace。

## 2. 关键性发现（Critical Findings）

### C1（关键）：账户挂起未生效 → 等同没有"封号"

- **位置**：`crates/veda-store/src/mysql.rs:2077-2087`（`get_api_key_by_hash`）；`auth.rs:91-99`、`auth.rs:160-176`（中间件）
- **症状**：`AccountStatus::Suspended` 字段在 schema、enum、查询里都存在，但鉴权路径**没有任何一处**校验它。
- **影响**：将来上线后被拉黑的账户依然可以用旧 token 操作 workspace；即使前端用户中心显示"已禁用"，后端实际畅行无阻。这是"以为已关门、实际门是开的"的安全洞。
- **建议修复**：在 `get_api_key_by_hash` SQL 里加 `INNER JOIN veda_accounts a ON a.id = veda_api_keys.account_id AND a.status = 'active'`；workspace key 路径同样补 join。然后再加一个针对挂起账户的回归测试。

### C2（关键）：`list_dentries_under(ws, "/")` 在 Reconciler / FilesTable 中全表扫描进内存

- **位置**：
  - `veda-core/src/store.rs:11-28`（trait 默认 impl 用 BFS 拉全部）
  - `veda-store/src/mysql.rs:897-904 → list_dentries_under_conn`：`SELECT ... WHERE workspace_id = ? AND path LIKE ? ESCAPE '\\' ORDER BY path`，**无 LIMIT**。
  - `reconciler.rs:432-441`、`reconciler.rs:447-479`：在 `list_mysql_file_ids` / `list_mysql_summary_entities` 里全表收集到 `Vec<Dentry>`。
  - `veda-sql/src/files_table.rs:222-232`：`MAX_SCAN_ENTRIES = 100_000` 是**事后**检查，已经把数据全 fetch 进内存才决定是否报错。
- **影响**：单个 workspace 文件数量大于内存可承载的量时，整个进程 OOM。`StorageStats` 不强制阈值；Reconciler 不分页。生产上这是 dam-burst：一个写得多的 workspace 直接打挂集群。
- **建议修复**：
  1. `list_dentries_under` 改流式 / 分页接口（cursor/keyset 分页）。
  2. Reconciler 在拉全表前先 `SELECT COUNT(*)`，超阈值就退化到分批 diff（按 path_prefix 切片）。
  3. FilesTable scan 在传给 store 之前就强制 LIMIT，DataFusion 端做完 limit pushdown。
  4. 全局加 `workspace.max_files` 软阈值，超阈值后写入失败。

### C3（关键）：CORS 默认 permissive

- **位置**：`crates/veda-server/src/main.rs:232-256`
  ```rust
  let cors = if cfg.allowed_origins.is_empty() {
      CorsLayer::permissive()
  } else { ... }
  ```
- **影响**：用户拷贝 `config/server.toml.example` 直接跑（README 里也是这么教的），就是 `allowed_origins = []` → 任何 origin 可调 API。如果 workspace token 被诱导泄漏到浏览器（用户开了 docs 站），任何站点都可以代用户调 fs API。
- **建议修复**：默认改成"未配置 allowed_origins 时拒绝跨域"，仅在 `dev_mode = true` 下保留 permissive。或者至少在文档里红字强调。

### C4（关键）：Worker `process_task` 失败时 `complete()` 已被各分支自调；但若 task 处理 panic（非 Err）则 `fail()` 不被调用 → `lease_until` 到期前一直是 processing

- **位置**：`worker.rs:62-97`
- **现象**：`for_each_concurrent` 里的闭包若 `process_task` panic（不可达？async fn 的 panic 会被 tokio 捕获到 join handle），`for_each_concurrent` 抛 panic 出来；`poll_once` 没有 catch_unwind。整个 worker tokio task 挂掉。
- **影响**：单条任务的脏数据（如空 payload）能击穿整个 worker。需要等系统操作员手动重启或外层 supervisor。
- **建议修复**：把 `self.process_task(&task)` 包一层 `tokio::task::spawn` + `.await` + 处理 `JoinError::is_panic` → 转成 `fail()`。或者更简单：`std::panic::AssertUnwindSafe(self.process_task(&task)).catch_unwind()`。

### C5（关键）：MySQL 时区假设未文档化、未在连接初始化时 `SET time_zone = '+00:00'`

- **位置**：`mysql.rs:432-450`（连接池构造）
- **现象**：所有 `naive_utc()` bound 进 sqlx 的 NaiveDateTime，sqlx 当成 server timezone 的字面量写入。如果 MySQL 服务器 `time_zone` 不是 UTC（CST、PST 等），`TIMESTAMP DEFAULT CURRENT_TIMESTAMP` 列存的是 server local time，但其它列存的是被 sqlx 解读为 server local 的 UTC 数值——错乱。
- **影响**：FS event 的 `id ASC` 排序可能错乱（id 是 AUTO_INCREMENT 递增，OK）；`available_at` / `lease_until` / outbox claim 的时间比较走 `UTC_TIMESTAMP()`，server timezone 不影响——这部分 OK。但 `created_at` 在跨时区显示给用户时，前端解读为 local 还是 UTC 取决于 server timezone。
- **建议修复**：在连接 URL 里加 `?time_zone=%2B00%3A00`，或在 pool 启动时执行 `SET SESSION time_zone = '+00:00'`，并在 `ARCHITECTURE.md` 里把 UTC 假设写成硬约束。

### C6（关键）：缺失速率限制 / 防滥用

- **症状**：
  - `POST /v1/accounts` 无任何 rate limit → 任意人可注册无限账户灌爆 `veda_accounts`。
  - `POST /v1/accounts/login` 无 limit → 凭证爆破不被遏制。Argon2 反爆破靠"单次验证慢"，但 1000 个并发请求一样能耗尽 CPU。
  - `POST /v1/sql` 无 limit、无超时 → 一条恶意 SQL 可绑死 worker。
  - `POST /v1/search` 触发 embed API 调用，按调用计费的 embed provider 会爆账单。
- **影响**：alpha 阶段对朋友共享或许 OK；任何"放到公网"的瞬间这是最先被打的入口。
- **建议修复**：用 `tower-governor` 或类似中间件加 per-IP / per-account 限流；`POST /v1/sql` 加查询超时（DataFusion 支持 `SessionConfig::with_target_partitions` 不限速但可加 `tokio::time::timeout`）。

### C7（中-高）：FUSE `setattr` 截断走 read-modify-write

- **位置**：`crates/veda-fuse/src/fs.rs:381-429`
- **现象**：`truncate(fd, 0)` → 写空 buffer。`truncate(fd, n)` 当 n>0 → 读全部 → 截到 n → 写回。
- **影响**：对 50MB 文件做 `truncate -s 100` 触发 50MB 下载 + 100 字节上传。Git/rsync 这种工具最常见的 setattr 模式。
- **建议修复**：服务端加 `PATCH /v1/fs/{path}/truncate?size=N` API；或者 setattr 路径直接调用一个新的 `fs.truncate(ws, path, n, base_rev)` 服务方法。

### C8（中-高）：Reconciler 在大 workspace 下退化为 N+1

- **位置**：`reconciler.rs:447-479`（`list_mysql_summary_entities`）
- **现象**：对每个 dentry 做一次 `get_summary_by_file` / `get_summary_by_dentry` 查询。100k dentries → 100-200k 次单行 SQL。每次 RTT 0.1ms 也是 10-20s 一次扫描。
- **建议修复**：把"哪些 file_id 有 summary"用 `SELECT file_id FROM veda_summaries WHERE workspace_id = ? AND status = 'ready'` 一次拉出来；和 dentry 集合做内存 join。同样 dir summaries。

## 3. 战略性问题（中长期会咬人）

### S8：Outbox / fs_events 表的"肿胀"风险
- `veda_fs_events` 有 retention sweep（默认 14 天）。OK。
- `veda_outbox` **没有 retention**——`completed` / `dead` 状态的行永久积累。一年后表行数 = 写入次数。`veda_outbox.status` 列没建索引，`claim()` 的 `WHERE status = 'pending' OR ...` 全表扫，但 `idx_claim (status, available_at)` 有覆盖，OK。但表本身无脑增长会拖慢备份、慢 SELECT、消耗 io。
- **建议**：加一个并行 sweep（已 completed N 天前的删掉）。已经有 fs_events 的退役模板可复用。

### S9：FUSE 写路径"全文重写"对大文件不友好
- `WriteHandle.buf: Vec<u8>` 持有整个文件内容。50MB 文件 open 一次 → 50MB 内存常驻。多个文件并发打开 → 内存爆。
- 服务端 `append` API 已有原子追加路径（incremental chunked append），但 FUSE 没有用——`flush_handle` 总是 `client.write_file(...)` 全量 PUT。
- **建议**：FUSE 端实现 dirty-region tracking，用 server 的 append 或部分写 API（需要新增 `PATCH` byte-range 写）。

### S10：搜索成本不可见
- 没有 `query_id` / `cost_unit` / `latency` 上报。一个用户跑 100 次搜索看不出 embed 调用花了多少钱。
- 已经 emit `veda_embed_total` 和 `veda_llm_total`，但没有 per-workspace 维度（这点 `reconciler.rs` 注释里说是为避免高基数 label 故意的——OK）。
- **建议**：加 per-account / per-workspace daily aggregated view（专门建 `veda_usage_daily` 表，从 fs_events + outbox + 中间件统计聚合）。是 SaaS 化的必备。

### S11：`storage_stats` 实际是"逻辑字节"而非"物理字节"
- 注释明确说 dedup 文件被 dentry 多算。
- 对计费：用户看到的 `total_bytes` 是 logical（用户视角的占用）；运维看到的存储表实际占用要从 `veda_files.size_bytes` SUM 出。
- **建议**：拆成两个字段：`logical_bytes`（per-user-view） 和 `physical_bytes`（dedup 后的实际）。否则运维成本核算会错配。

### S12：`Worker.handle_chunk_sync` 对一个 50MB 文件的内存压力
- 拉全文 → `String::with_capacity(file_size)`（50MB） → `semantic_chunk(...)` 切 chunks（≤50MB×2 因为 overlap）→ EMBED_BATCH=64，每 batch 又 clone 64 段 chunk content。
- 同时 `for_each_concurrent(batch_size = 10)` 跑 10 个文件 → 峰值约 1GB 常驻内存仅工作集。
- **建议**：把 `handle_chunk_sync` 做成"流式"：分段读 chunk、单段切 sem chunks、单段 embed、立即丢 buf。`semantic_chunk` 当前是全文一把切，需要先重构成 streaming。

### S13：`Cargo.lock` 在仓库里（OK）但 `target` 也在
- `target/` 在 `.gitignore` 里（确认了），但是 `ls -la` 显示根目录有 `target` 软链 / 真实目录。代码体积。
- 其实只是 build artifacts，OK。

### S14：单文件硬限 50MB 在 ARCHITECTURE 写"中型团队 10万-100万文件"
- 100 万 × 50MB = 50TB，MySQL `LONGTEXT` 撑得住但 `idx_checksum (workspace_id, checksum_sha256)` 这种索引在百万级以上做更新要排队。
- 没看到分库分表的入口。design.md 的"10万-100万文件 + 高可用"承诺需要重新评估单 MySQL 实例是否撑得住。

### S15：FUSE inode 表无大小上限
- `InodeTable.path_to_ino` HashMap 永久增长（除非 forget 把它删掉）。SSE 410 时只清 attr cache，不清 path↔ino 映射。
- 长时间挂载 + 频繁 ls 大目录 → 表项会无界增长。
- **建议**：加 LRU 淘汰（小心：被淘汰的 ino 必须先 fuser::Notifier 通知 kernel forget；否则 kernel 仍持有 stale ino 导致 ENOENT 风暴）。

## 4. 冗余 / 浪费

### W1：`MetadataStore` trait 的 default impl 在生产 store 中被覆写后是死代码
- 例如 `get_files_batch`、`get_summaries_by_file_ids`、`list_dentries_under` 都有 default impl + MysqlStore override。
- 这些 default impl 仅给 mock 用。把它们移到 mock-only helper 里更干净。

### W2：`SearchHit.path: Option<String>` 在 SearchResultItem 里被 `unwrap_or_default()` 转成 String
- `crates/veda-types/src/api.rs:106`：`path: h.path.unwrap_or_default()`
- 失败的路径解析会变成空字符串路径返回给客户端。CLI 显示就是"/"。**应该**保留为 `Option<String>` 让客户端区分"未知"和"根"。
- 修复：`SearchResultItem.path: Option<String>` + `#[serde(skip_serializing_if = "Option::is_none")]`。

### W3：`fs.rs` 大量 `unwrap` 在 Mutex lock 上
- 全文 `self.inodes.lock().unwrap()` 重复几十次。
- `Mutex` 中毒只在 panic 才发生；如果 fs 内部 panic，整个 FUSE 进程已经状态不一致。
- 但代码可读性受损。建议封装一个 `with_inodes<R>(|t| ...) -> R` 方法消除重复。

### W4：`WriteMeta.line_count` 在两处计算
- `compute_write_meta_blocking` 计算 line_count。
- `split_and_hash` 也计算 chunk-内 newlines + trailing partial logic。
- 两份逻辑都是对的但耦合度高，往后修改容易翻车。建议统一进 `split_and_hash` 的返回值。

### W5：`Cargo.toml` 工作区 `exclude` 导致 FUSE 测试不在 CI（详见 S7）
- 既然 fuse 是生产产出物之一，应该 include 或在 CI 单独跑。

## 6. 真心写得好的地方（保留 & 弘扬）

- **`worker.rs:259-267`** 的批 embed 注释：解释为什么不能用 `upsert_chunks`（带 trailing sweep）而要拆成 `upsert_chunks_only` + 末尾一次 `delete_chunks_above`。教科书级别的 partial-batch 安全分析。
- **`reconciler.rs:251-255`** 的 force_reembed 注释：把"watermark 正确但 Milvus 数据丢失"的悖论讲清了。
- **`auth.rs::AuthWorkspace`** 走 JWT-then-WorkspaceKey 的回退，配合 `read_only` 字段贯穿到 `FsUdfContext`。这种"权限维度沿调用链下穿"的设计很正。
- **`mysql.rs::try_insert_outbox_for_file`** 的 dedupe-only-against-pending 注释：解释为什么不能 dedupe `processing`（会丢更新）。这种 trade-off 讲得很透。
- **`fs.rs::write_file_once` 的 fast-path precheck**：客户端传 sha256，服务端能跳过整个 hash + write 路径直接返回 `content_unchanged: true`。CDN-like 的优化，给"git push 一个未变文件"这种场景非常友好。
- **`split_and_hash` 在单行超过 chunk_size 时延伸到下一个 `\n`**：保证 `start_line` 跨 chunk 唯一。这种"看起来像怪规则、实则维护核心 invariant"的代码注释做得很到位。
- **路径标准化拒绝 bidi 控制字符（`path.rs:54-71`）**：很多项目在这里翻车，用 RLO 字符来做钓鱼路径。本项目预先防御了。
- **`FsService::write_file` 的 `if_none_match_sha256` fast path（fs.rs:233-259）**：在拿到客户端 sha256 hint 时短路掉整个 DB 事务。

## 9. 关键设计决策记录（隐式 ADR）

> 这些决策散落在代码注释里，建议提炼为 `docs/INVARIANTS.md`。

1. **MySQL = control plane / Milvus = data plane**：分层一致性，Outbox + Reconciler 自愈，不做分布式事务。
2. **`upsert_chunks_only` + 单次 `delete_chunks_above`**：保证 batched embed 中途失败不开末尾空洞。
3. **`last_embedded_content_hash` 水印 + Reconciler 的 `force_reembed`**：把"已 embed 过"和"Milvus 实际有数据"解耦。
4. **Outbox `try_insert_outbox_for_file` 仅 dedupe `pending`**：dedupe processing 会让新内容被旧 in-flight task 覆盖。
5. **SSE 410 protocol 带 `current_max_id`**：客户端 list_dir 重同步时不漏 mid-recovery 事件。
6. **FUSE 用文件 row 的 `updated_at` 作为 mtime**：保证 `make`/`rsync -a`/`git status` 正确感知内容变化。
7. **路径校验拒绝 `:`、控制字符、bidi 控制字符**：远程路径 `:` 前缀 + 防 UI 钓鱼。
8. **`naive_utc()` bound 入 sqlx**：要求 MySQL server timezone = UTC（待显式化为 connect 后 SET）。
9. **CAS `revision` + If-Match header**：写入冲突在 HTTP 协议层暴露给客户端。
10. **`spawn_blocking` for SHA256 / chunk split**：CPU 重活不阻塞 tokio runtime worker。

## 结语

代码质量在 alpha 项目里属于第一档。注释和测试都看得出作者真的在思考，而不是拼凑。但项目当前的"工程化成熟度"和"承诺的目标场景"之间有显著 gap：

- README 写"successor to vecfs, rebuilt for multi-tenancy"——但 multi-tenancy 的最基本 ACL 校验（账户挂起）没生效。
- design.md 写"中型团队 5-50 人 / 10 万-100 万文件 / 高可用"——但 reconciler 在大 workspace 全表扫描 + N+1，单 MySQL 实例容量没规划。
- ARCHITECTURE.md 写"已实现"——但 collection schema 缺 reconciler、CORS 默认 permissive、无速率限制、无 rate limit。

**结论**：把"已实现"列表分级 — alpha (current) / production-ready / SaaS-ready 三档，并把当前优先级最高的 7-10 个 critical 项作为下一个 sprint 目标。代码本身已经做得对了，只差把"边界情况"和"对手存在"的视角补上。
