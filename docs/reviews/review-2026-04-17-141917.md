## Verdict

Sprint 2 is **functionally complete but not ship-ready**. Core table-function plumbing works (all 20 tests pass) and the code is well factored across `fs_table.rs` + `format/`. However, Sprint 1's recommended **read-only gate is half-done** — the flag was wired into `FsUdfContext` but the engine hardcodes `read_only: false` and the HTTP route never plumbs `auth.read_only`, meaning write UDFs are callable from any key regardless of permission. The table function itself does **eager I/O during planning**, so predicate / LIMIT pushdown buys nothing: `SELECT * FROM veda_fs('/logs/*.jsonl') LIMIT 1` reads every byte of every matching file before the LIMIT executes. The 10K-file guardrail fires *after* the memory spike it's meant to prevent. Fix C1 (security) and C2 (pushdown strategy) before merging; C3 and the strategic items can follow in Sprint 3.

## Critical Findings

### C1. Read-only gate is disconnected from auth — dead defense
- **File:** `crates/veda-sql/src/engine.rs:56-60`, `crates/veda-server/src/routes/sql.rs:17-24`
- **What:**
  `FsUdfContext { …, read_only: false }` is hardcoded in `VedaSqlEngine::execute`. `execute_sql` in `routes/sql.rs` extracts `auth: AuthWorkspace` but passes only `auth.workspace_id` to `sql_engine.execute(&auth.workspace_id, &req.sql)` — `auth.read_only` is never read. The `check_write_allowed` helper in `fs_udf.rs:96` will therefore **never trigger** in production.
- **Why it matters:**
  This was the explicit remediation for Sprint 1's Critical #2 ("read-only keys can still call `veda_write/append/remove/mkdir`"). The field got added, the check got added, the plumbing did not. A read-only API key today can still delete the workspace via `SELECT veda_remove('/')`. Worse, there are no tests for it, so the gap is invisible.
- **Recommendation:**
  1. Change `VedaSqlEngine::execute` signature to `execute(&self, workspace_id, sql, read_only: bool)`.
  2. Pass `auth.read_only` from `routes/sql.rs:22-24`.
  3. Add an integration test: mount a read-only auth, execute `SELECT veda_write('/x', 'y')`, assert it returns a permission error.

### C2. Table function does eager I/O at plan time — no pushdown, no LIMIT shortcut
- **File:** `crates/veda-sql/src/fs_table.rs:36-89`
- **What:**
  `VedaFsTableFactory::call` synchronously runs `list_dir_recursive`, `read_file`, and `parse_file` **before** constructing the `TableProvider` — it wraps everything in a `MemTable` of fully-materialized `RecordBatch`es. There is no `TableProvider::scan` override, so DataFusion's projection / predicate / limit pushdown does nothing useful: the data is already in memory.
- **Why it matters:**
  - `SELECT line FROM veda_fs('/logs/*.jsonl') LIMIT 1` reads every byte of every matching file.
  - `SELECT line FROM veda_fs('/huge.csv') WHERE _line_number = 1` reads and parses the whole CSV.
  - Planner cost estimates are meaningless — all work is done in `call()`.
  - `EXPLAIN` on a query using `veda_fs()` will already have paid the full I/O cost. Users expect planning to be cheap.
- **Recommendation:**
  Short-term: document the limitation prominently in the plan and README, and cap the size defenses much more aggressively (see C3). Medium-term: replace `MemTable` with a custom `TableProvider` that defers reads to `scan()` and honors `limit`. Even a naive implementation that streams files one at a time and stops when `limit` is satisfied would fix the LIMIT path.

### C3. 10K-file glob guardrail triggers *after* the memory spike it's supposed to prevent
- **File:** `crates/veda-sql/src/fs_table.rs:65-73`, `crates/veda-core/src/service/fs.rs:304-328`
- **What:**
  The `Glob` branch calls `fs.glob_files(&ws, &path)` which internally calls `list_dir_recursive`, BFS-loading **every `Dentry` under the fixed-prefix subtree** into a `Vec<Dentry>`. Only then does `fs_table.rs` check `matching.len() > MAX_GLOB_FILE_COUNT`. If a workspace has 1M files under `/`, the pattern `/**/*.log` allocates 1M dentry structs before the check rejects the query.
- **Why it matters:**
  The limit exists to prevent denial-of-service; it doesn't. An attacker with a read key can still OOM the server by crafting a glob over a large subtree.
- **Recommendation:**
  Push the count limit down into `list_dir_recursive` / `glob_files`: accept a `max_count: usize` parameter, abort mid-walk when exceeded. Same applies to total bytes: make the byte-budget check streaming, inside the BFS, rather than after the fact in `read_glob_files` (which at least bails mid-loop — good — but the file list is already materialized).

## Strategic Concerns

### S1. Glob fixed-prefix extraction walks the entire workspace for common patterns
- **File:** `crates/veda-core/src/service/fs.rs:825-839`
- **What:** `glob_fixed_prefix("/*.txt")` returns `"/"`, so `list_dir_recursive` walks the whole workspace just to find top-level `.txt` files. Same for `/**/*.log`, `/?foo.json`, and any pattern that starts with a wildcard at depth 1.
- **Long-term framing:** This is fine at current scale (small workspaces), painful at 10K+ files, and a direct cliff at 1M. Combined with C3, a non-admin user can trigger whole-workspace scans cheaply. A future `dentries(workspace_id, path_prefix)` SQL index query would be much cheaper than BFS in Rust.
- **Recommendation:** Defer — but track it. When workspace sizes grow, move glob filtering into SQL `WHERE path LIKE 'prefix%'` against `dentries` and evaluate the tail pattern in Rust.

### S2. Empty-glob fallback uses PlainText schema regardless of extension
- **File:** `crates/veda-sql/src/fs_table.rs:78-82`
- **What:** When a glob matches zero files, the fallback schema is `format::parse_file("", "/empty", &FileFormat::PlainText)`, i.e. `(_line_number, line, _path)`. If the pattern was `/logs/*.csv`, downstream SQL written against CSV columns now fails with a schema mismatch that the user can't easily explain ("where did `_line_number`/`line` come from?").
- **Recommendation:** Derive the fallback schema from `detect_format(pattern)` so an empty CSV glob still returns the CSV-shaped empty batch. Or use a stable "empty" sentinel schema and document it.

### S3. Dir listing issues N+1 stat calls
- **File:** `crates/veda-sql/src/fs_table.rs:109-151`
- **What:** `build_dir_listing_batch` calls `fs.stat(ws, &d.path)` once per dentry, each of which hits MySQL. For 1000 entries under a directory this is 1001 synchronous queries through `block_on`.
- **Why it matters:** This is the "listing" path users will hit interactively — slow listings are the first impression. And every call holds up a worker thread via `block_in_place`.
- **Recommendation:** `Dentry` already contains `updated_at`; `size_bytes` needs the file record. Add a `MetadataStore::stat_batch(ws, &[paths])` that does one joined query, or extend `list_dentries` to include file size. Don't parallelize — reduce the query count.

### S4. CSV schema is inferred per-file with no cross-file validation
- **File:** `crates/veda-sql/src/format/csv.rs:20-23`, `crates/veda-sql/src/fs_table.rs:183-213`
- **What:** `arrow_csv::Format::infer_schema` samples 100 rows per file. In `read_glob_files`, the second and subsequent CSVs inherit the first batch's schema (`batches[0].schema()`) via `MemTable`. If file #2 happens to have differently typed columns (or reordered columns, or extra columns), the `MemTable::try_new` call will return a cryptic "schema mismatch" error — not a user-friendly diagnostic.
- **Recommendation:** Before appending each CSV batch, compare schemas explicitly and emit a targeted error: "file `x.csv` has column `age` of type `Float64` but file `y.csv` had `Int64`." Or, push schema unification into `format::csv` and promote incompatible types to `Utf8`.

### S5. JSONL silently drops invalid lines
- **File:** `crates/veda-sql/src/format/jsonl.rs:24-27`
- **What:** Lines that fail `serde_json::from_str` are skipped with no count, no warning, no surfaced column.
- **Why it matters:** This is the exact failure mode SQL-on-logs users care about. A corrupt line becomes invisible. It also makes `COUNT(*)` non-deterministic across file reads if encoding changes.
- **Recommendation:** Either (a) surface an `_is_valid` boolean column and keep the raw line, or (b) add a `veda_fs_errors` stats counter. At minimum, log at `warn!` level when a line is dropped, with the path and line number.

### S6. Glob mode documents `[...]` character classes but doesn't support them
- **File:** `crates/veda-sql/src/fs_table.rs:100`, `crates/veda-core/src/service/fs.rs:847-`
- **What:** `detect_mode` treats `[` as a glob trigger, but `glob_match_inner` only implements `*`, `**`, `?`. A pattern like `/log/[ab].txt` routes to the Glob branch and then silently matches zero files (the literal `[ab].txt` won't appear on disk).
- **Recommendation:** Either implement `[...]` classes (small extension) or remove `[` from `detect_mode`'s trigger set so the path is treated as a literal FileRead.

## Redundancy & Waste

### W1. Duplicate constants: `MAX_READ_SIZE` vs `MAX_SINGLE_FILE_READ` (both 50 MB)
- **Files:** `crates/veda-sql/src/fs_udf.rs:14`, `crates/veda-sql/src/fs_table.rs:19`
- **What:** Two separate `50 * 1024 * 1024` constants protecting the same operation (read a single file).
- **Recommendation:** Hoist to `crate::limits` (or a `const` in `fs_udf`) and import in both.

### W2. Duplicate schemas: `text::text_schema()` and `jsonl::jsonl_schema()` are identical
- **Files:** `crates/veda-sql/src/format/text.rs:7-13`, `crates/veda-sql/src/format/jsonl.rs:7-13`
- **What:** Both define `(_line_number: Int64, line: Utf8, _path: Utf8)`. The JSONL path doesn't parse the JSON — it just validates and stores the raw line. So the schemas are the same shape today.
- **Recommendation:** Pick one owner (`format::line_schema()`), import in both. If JSONL later parses into typed columns, the two will diverge naturally — but right now they're duplication.

### W3. `build_dir_listing_batch` has an unused error branch that falls back to dentry mtime
- **File:** `crates/veda-sql/src/fs_table.rs:134-137`
- **What:** If `fs.stat` fails, code uses `d.updated_at` for mtime and null for size. But `fs.stat` only fails if the entry was deleted mid-listing — an extremely rare race. The branch exists, but there's no test and the fallback is inconsistent with the success path.
- **Recommendation:** Either document the race or skip the entry (drop from listing). Silently emitting a row with null size and stale mtime is worse than either.

### W4. CSV empty-content check duplicated
- **File:** `crates/veda-sql/src/format/csv.rs:9-16` and `43-51`
- **What:** Two separate "return empty RecordBatch" branches: one when `content.trim().is_empty()` (before infer), another when `batches.is_empty()` (after reader yields nothing). The first returns a fixed "line-oriented" schema; the second uses the inferred schema. They're inconsistent — a CSV with only a header produces one shape, a truly empty file produces another.
- **Recommendation:** Always infer on a minimal header, return a consistent schema. Or delete the first branch and let the reader naturally yield zero batches.

## What's Good

- Clean module split: `fs_table.rs` handles mode dispatch, `format/` owns parsing. Swapping a format requires touching one file.
- `FsMode` enum + `detect_mode` is the right abstraction — cheap to read, cheap to extend with future modes (Parquet, SQLite).
- Sprint 1's Critical #1 (per-row scalar UDF) was genuinely fixed; `get_string_values` correctly broadcasts scalars and unwraps arrays, and the two new tests (`udf_column_arg_veda_read`, `udf_column_arg_veda_exists`) actually exercise the multi-row path.
- Guardrails exist everywhere the spec calls for them (50 MB / 100 MB / 10K files), even if placement is wrong (C3). The intent is right, the wiring needs tightening.
- `block_on` correctly uses `block_in_place` on multi-thread runtimes — a subtle footgun that was avoided.
- Test coverage grew from 13 to 20 tests; all new SQL surface has at least one happy-path test.

### Key Decisions Observed

- **Table function is registered via `ctx.register_udtf`, not as a catalog table.** Correct choice for per-query path binding, but it means every query re-plans the table — the eager I/O cost (C2) is paid per query, not amortized.
- **`MemTable` as a deliberate shortcut.** The plan cost is acceptable for Sprint 2's milestone, but this is a load-bearing decision: the whole plan-time-I/O problem (C2) flows from it. When Sprint 3/4 demands streaming, this becomes the first refactor target.
- **Glob syntax is hand-rolled** rather than using the `glob` crate. Fine for now (small surface area, no external deps), but explains S6 (no character classes) and means any future syntax extension lives in `fs.rs`. Reconsider if complexity grows.
- **CSV/TSV use `arrow_csv` directly**, JSONL and text use hand-built builders. Inconsistent but pragmatic — `arrow_json` would complicate the "line-oriented" semantics users expect for log files.
- **Format detection is extension-only** (`.csv` / `.tsv` / `.jsonl` / `.ndjson`). No content sniffing, no MIME hint from `FileInfo`. Simple and predictable; the tradeoff is that `foo.log` holding JSON lines falls back to PlainText.
