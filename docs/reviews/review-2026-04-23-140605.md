**Date:** 2026-04-23
**Reviewer:** Codex 5.3
**Scope:** 当前主分支实现（全仓 `crates/*` 逐模块静态审查 + `cargo test` 基线验证）

**Files Analyzed:** 30+（核心实现路径）

## Verdict

当前实现整体可运行（`cargo test` 全部通过，忽略项除外），但仍存在若干高风险正确性与一致性问题，主要集中在 `append` 行数计算、Milvus stale chunk 清理、以及 ref_count 下界约束。若直接上线到高并发或长时间运行环境，可能出现“测试难以覆盖、线上才暴露”的数据一致性偏差和检索污染。

## Critical Findings

### 1) 增量 append 的行数计算存在语义错误
- **File:** `crates/veda-core/src/service/fs.rs`
- **What:** `before_line_count = file.line_count - last.line_count` 把“文件总行数”减去“最后 chunk 的换行数”，语义不一致；当最后 chunk 不以换行结束时会高估行数。
- **Why it matters:** `line_count` 被用于 `read_file_lines` 的边界裁剪，可能导致按行读取行为错误。
- **Recommendation:** 以 `last.start_line - 1`（或等价严格语义）计算 tail 之前行数，补充“chunked 且末尾无换行”测试。

### 2) Milvus upsert 后 stale chunk 删除失败被静默吞掉
- **File:** `crates/veda-store/src/milvus.rs`
- **What:** `upsert_chunks` 中 delete 调用结果被 `let _ = ...` 丢弃。
- **Why it matters:** 新数据写入成功但旧 chunk 清理失败时，会残留幽灵向量，搜索召回与实际文件内容不一致。
- **Recommendation:** delete 失败必须返回错误（进入 outbox 重试）或显式重试并记录告警，不允许静默忽略。

### 3) ref_count 可减到负数
- **File:** `crates/veda-store/src/mysql.rs`
- **What:** `decrement_ref_count` 未限制下界，`current - 1` 后直接写回。
- **Why it matters:** 负 ref_count 会触发错误的“可删除”判断，可能误删文件内容/向量索引。
- **Recommendation:** 改为原子 `UPDATE ... SET ref_count = ref_count - 1 WHERE id = ? AND ref_count > 0`，并在失败时返回明确错误。

## Strategic Concerns

### 1) 认证失败与存储失败混淆为同一 401
- **File:** `crates/veda-server/src/auth.rs`
- **What:** `map_err(|_| auth_err())` 将 auth_store 的真实内部错误也映射成 unauthorized。
- **Why it matters:** 线上故障会被误判为权限问题，排障与告警失真。
- **Recommendation:** 区分“凭证无效(401)”与“依赖异常(5xx/503)”，并记录结构化日志。

### 2) 登录错误语义与 HTTP 码不匹配
- **File:** `crates/veda-server/src/routes/account.rs`, `crates/veda-server/src/error.rs`
- **What:** 登录凭证错误走 `PermissionDenied -> 403`。
- **Why it matters:** 与常见 auth 语义不一致，影响客户端与网关策略。
- **Recommendation:** 引入 `Unauthorized/InvalidCredentials` 并映射 401。

### 3) Workspace key 权限参数宽松回退为读写
- **File:** `crates/veda-server/src/routes/account.rs`
- **What:** 除 `"read"` 外全部回退为 `ReadWrite`。
- **Why it matters:** 拼写/大小写错误可能发放超权限 key。
- **Recommendation:** 严格白名单校验（`read` / `readwrite`），未知值返回 400。

### 4) CORS 默认全开放
- **File:** `crates/veda-server/src/main.rs`
- **What:** 使用 `CorsLayer::permissive()`。
- **Why it matters:** 公网部署时扩大跨域攻击面。
- **Recommendation:** 改为配置化 allowlist，默认最小权限。

## Redundancy & Waste

### 1) SQL 文件 glob 路径绕过单文件读取上限
- **File:** `crates/veda-sql/src/fs_table.rs`, `crates/veda-sql/src/fs_udf.rs`
- **What:** `read_single_file` 使用 `bounded_read_file`，但 `read_glob_files` 直接 `fs.read_file`。
- **Why it matters:** 同类读取路径上限策略不一致，可能在 glob 下触发更高内存压力。
- **Recommendation:** glob 逐文件读取也统一走 bounded/stat 限制逻辑。

### 2) Collection SQL 扫描未利用 filters
- **File:** `crates/veda-sql/src/collection_table.rs`
- **What:** `scan(..., _filters, ...)` 忽略过滤条件，固定 query + 后过滤。
- **Why it matters:** 大集合查询成本偏高，扩展性差。
- **Recommendation:** 能下推则下推；不能下推时至少在文档和错误信息中说明。

## Performance Bottlenecks

### 1) Range 读取实际是整文件读取后切片
- **File:** `crates/veda-core/src/service/fs.rs`, `crates/veda-server/src/routes/fs.rs`
- **What:** `read_file_range` 先 `read_file` 全量，再按 offset/length 截取。
- **Impact:** 大文件 range 请求仍是 O(file_size) CPU/内存。
- **Fix:** 增加基于 chunk/offset 的分段读取路径，避免整文件 materialize。

### 2) FUSE 读缓存命中仍复制整块内存
- **File:** `crates/veda-fuse/src/fs.rs`
- **What:** `cache_get` 命中后 `to_vec()` 克隆。
- **Impact:** 高频读场景增加内存带宽与 CPU 开销。
- **Fix:** 改为共享 buffer（`Arc<[u8]>`/`Bytes`）或 slice 视图。

### 3) ReadCache LRU 驱逐是 O(n) 线性扫描
- **File:** `crates/veda-fuse/src/cache.rs`
- **What:** `evict_lru` 每次 `min_by_key` 遍历全表。
- **Impact:** 大缓存条目下写入/替换退化明显。
- **Fix:** 使用 O(1)/O(log n) 的 LRU 结构（如链表+哈希或成熟 LRU 容器）。

### 4) Worker shutdown 响应不及时
- **File:** `crates/veda-server/src/worker.rs`
- **What:** `poll_once` 内部 sleep/批处理循环期间不监听 shutdown。
- **Impact:** 停机延迟，可能拖慢滚动发布。
- **Fix:** 用 `select!` 包裹 sleep 与 shutdown，批处理中定期检查取消信号。

### 5) CLI 与 Server 默认端口不一致导致冷启动体验差
- **File:** `crates/veda-cli/src/config.rs`, `crates/veda-server/src/config.rs`
- **What:** CLI 默认 `9009`，Server 默认 `3000`。
- **Impact:** 新环境首次使用易“连接失败”。
- **Fix:** 统一默认端口并同步文档。

## Documentation Gaps

- `docs/design.md` 仍描述 `WS /v1/ws/events`，当前实现是 `GET /v1/events`（SSE），文档与实现不一致。
- 部分设计文档对“文本-only 约束”与 FUSE 行为边界（UTF-8 前提）未明确，容易引发使用预期偏差。

## What's Good

- `cargo test` 基线稳定，核心单测覆盖广（fs/sql/type/core）。
- FS 写路径引入了 CAS（If-Match）与去重短路（If-None-Match sha256），正确性方向好。
- 路径规范化与 Unicode 处理相对完整，具备较好输入边界防护。

### Key Decisions Observed

- 当前系统明确偏向“文本内容 + 最终一致 outbox + 异步索引更新”的架构；这使业务层简单，但需要更强的一致性防护（删除失败处理、ref_count 守护）。
- SQL 层采取“快速可用优先”（拉取后处理）的实现风格，后续扩展阶段应逐步补齐 pushdown 与资源上限一致性。
- FUSE 实现偏功能优先，缓存和读路径仍有可观的性能优化空间。
