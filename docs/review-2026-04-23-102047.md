# Deep Review Report

**Date:** 2026-04-23
**Reviewer:** Claude Opus 4.7
**Scope:** 全仓 bug / 设计缺陷 / 性能瓶颈审查（veda-core / veda-store / veda-sql / veda-server / veda-fuse / veda-pipeline）
**Commits Reviewed:** main 分支当前状态（含 3 处未提交修改）
**Files Analyzed:** 27

## Verdict

整体架构合理，但实现层有若干**会在生产规模下崩掉**的热点：`read_file_range` 根本没走 range、`append` 路径会把整个文件加载进内存重新哈希、`veda_dentries.path` 没有可供 `LIKE` 使用的索引、目录级操作（删除/重命名/递归列表）在单事务里做 N 次 UPDATE/DELETE。在离线/单用户场景下能跑，但只要触达中等规模（~10 万文件 / >10 MB 文件），延迟和锁竞争会迅速劣化。另有若干正确性 bug（FUSE 的 `setattr` / `create` 无 revision 校验、目录级 FS event 丢失、login 无限生成 api_key），建议在继续功能开发前先吃掉这批。

## Critical Findings

### 1) `read_file_range` 先整文件读到内存再切片，Range 语义完全失效
- **File:** `crates/veda-core/src/service/fs.rs:439-456`
- **What:** `read_file_range` 内部调用 `self.read_file(...)`（加载全部内容到 `String`），再 `bytes[offset..end].to_vec()`。
- **Why it matters:** FUSE 客户端对 >1 MB 的文件会走 `read_range`（见 `veda-fuse/src/fs.rs:390`）。每次 4 KB 的 kernel page read 都把 50 MB 的文件从 MySQL 全量拉出（`get_file_chunks` 无过滤）→ 解码为 UTF-8 → 复制到 `Vec<u8>`。放大系数 ≈ 12 000×。生产场景下单次 range read 大概率在 30 s 内打爆 MySQL 流量 / tokio 堆。
- **Recommendation:** 推到 `MetadataStore` 层：加 `get_file_byte_range(file_id, offset, length)`。chunked 存储下按 `byte_offset` 定位起始 chunk（`byte_len` 可累加出 offset 边界，配合现有 `idx_line_lookup` 再加一个字段或新建 offset 索引），inline 存储下用 `SUBSTRING(content, ?, ?)`。同时修正 `parse_range_header` 对 `bytes=start-` 使用 `u64::MAX` 长度的写法（`routes/fs.rs:229-230`），避免 `saturating_add` 之后仍然触发全量读。

### 2) `append_file` 增量路径会把已有全部 chunk 内容拉回内存重算哈希
- **File:** `crates/veda-core/src/service/fs.rs:963-988`, `crates/veda-core/src/service/fs.rs:174-207`
- **What:** 增量 append 声称只重切 tail，但 `compute_append_meta_blocking` 为了算新 sha256 需要把所有 `chunk_index < last_chunk_idx` 的内容再 feed 一次 hasher（`for c in &before { hasher.update(c.content.as_bytes()); }`），而 `all_chunks = tx.get_file_chunks(fid, None, None).await?` 读取了完整内容。
- **Why it matters:** 对一个 49 MB 文件 append 1 KB，实际会从 MySQL 读 49 MB → 在 tokio 线程上 copy 49 MB 到 `Vec<FileChunk>` → 在 blocking pool 上再跑 49 MB SHA-256。标称 "incremental"，实际复杂度 O(文件大小)。
- **Recommendation:** 给 `veda_files` 表加 `prefix_sha256_state`（SHA-256 中间状态序列化，≈ 104 字节）或者在 `veda_file_chunks` 里冗余一个 `cumulative_sha256_before_this_chunk`。append 时只需加载最后一个 chunk、恢复 hasher 状态、喂入 tail。也可退而求其次：直接去掉 "content_unchanged" 检测，append 不计算完整 sha256（append 场景极少发生整体重复），只更新 tail 对应的 `chunk_sha256`，整体 sha256 放到 lazy checksum / 由 Reconciler 校对。

### 3) `veda_dentries` 上所有 `path LIKE` 查询是全表扫
- **File:** `crates/veda-store/src/mysql.rs:371-384`（建表）、`mysql.rs:544-566`（`list_dentries_under`）、`mysql.rs:981-1004`（`delete_dentries_under`）、`mysql.rs:826-838`（`query_fs_events`）
- **What:** `veda_dentries` 只有 `UNIQUE (workspace_id, path_hash)` 和 `INDEX (workspace_id, parent_path(255))`。但 `list_dentries_under`、`delete_dentries_under` 和 `FilesTable::scan` 的前缀下推都走 `path LIKE '/prefix/%'`。`path_hash` 是 SHA2，无法走 LIKE；`parent_path(255)` 也用不上 `path` 列上的前缀。→ 每次递归列表、目录删除、递归 rename、SQL `files` 表扫，都在 MySQL 里做全索引扫/全表扫。
- **Why it matters:**
  - `list_dir_recursive` / `glob_files` / `FilesTable::scan` 在 workspace 规模到 10 万 entry 以上基本不可用（秒级）。
  - `delete` 一个大目录在事务里跑一条无索引的 `DELETE ... WHERE path LIKE '...'`：不仅慢，InnoDB 还会给所有扫过的行（哪怕不匹配）加 next-key lock，阻塞并发的根目录写入。
- **Recommendation:** 给 `veda_dentries` 加 `INDEX idx_ws_path_prefix (workspace_id, path(255))`（path 列前缀索引），MySQL 的 `LIKE 'fixed/%'` 能用上它（leftmost prefix）。或者换思路：显式存 `path_parts`（array 或打平的逐级 ancestor），查询用 ancestor 等值而非 LIKE。顺便 `veda_fs_events` 的 `path LIKE` 查询同理——小流量下影响较小，但值得同步处理。

### 4) 目录级 FS event 丢失：delete/rename 只发 1 条根路径事件
- **File:** `crates/veda-core/src/service/fs.rs:661-712`（`delete`）、`fs.rs:1036-1102`（`rename`）
- **What:** 删除一个目录时，代码收集了 `children` 做 ref_count 清理，但 `insert_fs_event` 只对顶层 `norm` 插入一条事件。同理 rename 只对新路径发 `Move`。
- **Why it matters:** FUSE 依赖 `/v1/events` SSE 驱动 attr/read cache 失效（`veda-fuse/src/sse.rs` + `fs.rs:185-186`）。删了 `/data/` 目录，FUSE 端会继续把 `/data/sub/x.txt` 的缓存当有效；客户端调 `ls` 时命中 LRU 里的旧内容，返回已不存在文件的字节。典型"幽灵文件"bug。
- **Recommendation:** 在删除分支里给每个 child 也 `insert_fs_event(Delete)`，rename 分支里给每个被改名的子项补 `Move` 事件。批量写一条 `INSERT ... VALUES (...), (...)` 即可。

### 5) FUSE `setattr`(truncate) 与 `create` 绕过了 revision 校验
- **File:** `crates/veda-fuse/src/fs.rs:242-287`（setattr）、`fs.rs:435-459`（create）、`fs.rs:448`（`write_file(path, b"", None)`）
- **What:** `setattr` 做 truncate 是 read → resize → `write_file(..., None)`；`create` 直接 `write_file(path, b"", None)`，都没有传 `expected_revision`。
- **Why it matters:** truncate 与 create 都是 "应该破坏性修改但保留对并发写入的感知" 的操作。现在的实现：另一个客户端在 read 和 write 之间写了新内容——FUSE 用 `None` 写覆盖，无声丢数据。`create` 同理：kernel 认为 "不存在才 create"，但在网络 FS 里 lookup→create 之间远端可能已创建并写入，当前实现会把它覆盖为空。
- **Recommendation:** `setattr(size=...)` 先 `stat` 拿 revision，再 `write_file(..., Some(rev))`；`create` 对 O_EXCL 场景强制 revision=0（"必须不存在"），非 O_EXCL 但有冲突时按 EEXIST 返回。

### 6) `login` 每次调用都追加一个永不失效的 API key
- **File:** `crates/veda-server/src/routes/account.rs:84-120`
- **What:** 登录流程创建 `vk_...` 键并 insert 到 `veda_api_keys`，无 `expires_at`，无旧键失效。
- **Why it matters:**
  1. 安全面：任意一次历史登录泄露的 key 永远有效，无吊销路径（`revoke_api_key` 只按 id，用户不知道 id）。
  2. 存储面：人类用户 × 20 次登录 / 月 × 12 月 ≈ 240 行/人；脚本化重复登录会把 `veda_api_keys` 撑爆。
  3. 并发面：`get_api_key_by_hash` 只校验 `status = 'active'`，不限制每账号同时活跃 key 数量。
- **Recommendation:** login 应当返回短期 JWT（沿用 `create_jwt`）而不是持久 API key；`/v1/accounts/login` 改为不写数据库。持久 API key 走 `/v1/accounts/keys` 显式创建、带 `expires_at`。

### 7) `insert_file_chunks` 循环单行 insert，事务内 N 次 round trip
- **File:** `crates/veda-store/src/mysql.rs:1187-1207`
- **What:** 对每个 `FileChunk` 单独执行一次 `INSERT ... VALUES (?, ...)`，写入放在同一个 `MetadataTx` 事务中。
- **Why it matters:** 一个 50 MB 文件的首次写入会产生 200 个 chunk → 200 次 round trip。对 MySQL 的"事务内 N 次小 insert"是典型抗性不好的模式：每条 statement 都穿越网络、每条触发 redo log 刷盘（除非 `innodb_flush_log_at_trx_commit=2`）。实测同局域网下单事务 200 条小 INSERT ≈ 300 ms。
- **Recommendation:** 改为 `INSERT ... VALUES (...), (...), ...` 一条语句批量写（chunk 数在 200 级别，远未到 `max_allowed_packet` 上限）。建议封装成 `insert_file_chunks_batch`，保持 trait 签名不变（内部分批即可）。

### 8) 目录 rename 对每个子项发一条 UPDATE
- **File:** `crates/veda-core/src/service/fs.rs:1081-1093`
- **What:** `rename` 目录时 `for child in &children { tx.rename_dentry(...).await?; }`，每个 child 一条 UPDATE。
- **Why it matters:** 改名一个 1 万文件的目录 = 1 万次 UPDATE。加上 Critical 3 的 LIKE 扫表拉出 children，整个操作对 MySQL 是分钟级的长事务，并且整段时间持有 dentry 行级锁 → 并发写全部阻塞。
- **Recommendation:**
  ```sql
  UPDATE veda_dentries
     SET path = CONCAT(?, SUBSTRING(path, ?)),
         parent_path = CONCAT(?, SUBSTRING(parent_path, ?))
   WHERE workspace_id = ?
     AND (path = ? OR path LIKE CONCAT(?, '/%'))
  ```
  一条语句搞定。注意 `path_hash` 是 STORED generated column，MySQL 会自动重算。

### 9) SSE `/v1/events` 每个订阅每秒一次 MySQL 轮询
- **File:** `crates/veda-server/src/routes/events.rs:27-64`
- **What:** 单用户一次长连接 → 固定 `POLL_INTERVAL = 1s` 轮询 `query_fs_events`，无 backoff、无多连接共享。
- **Why it matters:** 100 个 FUSE 挂载同时连上 = 100 QPS 常驻流量打在 `veda_fs_events` 上。workspace 隔离意味着无法通过合并查询优化；`SKIP LOCKED` 也用不上（只读）。
- **Recommendation:** 进程内维护"watch events"的内存广播（`tokio::sync::broadcast`），workspace 写事件路径直接塞入 channel；SSE handler 只订阅 channel 不轮询 MySQL。MySQL 仅作为"冷启动 cursor 追赶"的回放源。顺带：轮询失败时应该指数退避而不是照样 1 s 再来（`Err` 分支当前只 warn）。

### 10) Milvus 每次写入都 flush
- **File:** `crates/veda-store/src/milvus.rs:415-424`、`milvus.rs:436-444`、`milvus.rs:620-626`
- **What:** `upsert_chunks` / `delete_chunks` / `insert_collection_rows` 末尾都调用 `/v2/vectordb/collections/flush`。
- **Why it matters:** Milvus 官方明确 flush 是重操作（触发 segment sealing + 持久化），推荐仅用于"确实需要立即可见"的场景。Worker 每处理一个 outbox 任务都 flush 一次——当 outbox 堆积时，Milvus 集群会被 flush 风暴打趴。
- **Recommendation:** 去掉写入路径上的 flush。保留在 `init_collections` / 测试显式 flush 工具里。Milvus 的 ANN 搜索会从内存（growing segment）搜到新写入的向量，可见性通常不是问题；如果真要强一致，Worker 里周期性 flush（比如每分钟一次或每 N 个任务一次）。

## Strategic Concerns

### 1) `FsService` 大量热路径仍然在单个大事务里串行
- **File:** `crates/veda-core/src/service/fs.rs:280-384`（`write_file_once`）、`fs.rs:661-713`（`delete`）、`fs.rs:1036-1103`（`rename`）
- **What:** 写入 / 删除 / rename 都是 "拿锁 → 一堆 query → commit" 单线程串行。
- **Why it matters:** 业务正确性靠 `MetadataTx::FOR UPDATE` 维持，但一旦某个 query 慢（例如 Critical 3、Critical 7），整个 workspace 的写并发都被卡住。典型场景：A 用户做 `delete /big-dir` 花 3 分钟，同一 workspace 的任何 `write_file` 都会等 dentry 锁。
- **How to apply:** 考虑把"真正需要事务"的原子单位缩到最小：file 元数据 + 当前 dentry。content 写入（file_contents / file_chunks）改为幂等的两阶段：先写 content 到 staging 表或直接写 + 拿 revision-based CAS，再提交 dentry 切换。路径锁的粒度也可以细化到 path 而不是整个事务占用多行。

### 2) `veda-core::store::MetadataStore::get_files_batch` 的 default 实现是 N+1
- **File:** `crates/veda-core/src/store.rs:29-37`
- **What:** 默认 impl `for id in file_ids { self.get_file(id).await }`；MySQL 后端已 override 为一条 `IN (...)` query。
- **Why it matters:** trait 设计上"可选优化"而非强制，换个 backend 或者测试 mock 忘了 override 就退化。`FilesTable::scan` 对此调用（`files_table.rs:247`），影响所有 SQL 文件查询。
- **How to apply:** 把这个方法设为 trait required（无 default），或直接删掉改成外层 `join_all(ids.map(|i| store.get_file(i)))`——这样后端不会默默退化。

### 3) 路径规范化宽度不够：允许了控制字符 / unicode 双向覆写
- **File:** `crates/veda-core/src/path.rs:38-50`
- **What:** `validate_segment` 只拒 `\0` 和 `:`。
- **Why it matters:** 路径里可以塞换行、回车、RLO (U+202E)、零宽连字符等；在 CLI/TUI/浏览器里可以做路径欺骗（"`/safe.txt`" 和 "`/safe\u{202E}txt.exe`" 视觉相同）。也会污染日志（换行可以伪造额外日志行）。
- **How to apply:** 拒绝 `ASCII < 0x20`、DEL、U+2028/U+2029、BIDI control chars；对 Windows 的客户端考虑同时拒 `<>?|"*`。NFC 已做，是好事；再加一道 `segment.chars().all(|c| !c.is_control() && !is_bidi_control(c))`。

### 4) `decrement_ref_count` 是两段式（UPDATE + SELECT），有轻微 TOCTOU
- **File:** `crates/veda-store/src/mysql.rs:1104-1118`
- **What:** 先 `SET ref_count = ref_count - 1`，再 `SELECT ref_count`，以此决定 "是否 orphan 可以删 content/chunks"。
- **Why it matters:** 两次 UPDATE 并发且"读 ref_count"未加 FOR UPDATE 的话，两个并发 decrement 都可能看到 remaining=0 并都去 `delete_file_content`。当前事务层面的行锁让这个场景实际上被序列化（UPDATE 会排队），但依赖行为不写明约束——将来换存储或者有人把事务隔离改成 READ COMMITTED 就会出错。
- **How to apply:** 改成 `UPDATE ... WHERE id = ? AND ref_count > 0 RETURNING ref_count`（MySQL 8.0.21+）或 `SELECT FOR UPDATE` 然后计算。现在事务里已有 `get_file FOR UPDATE` 的 pattern，直接复用。

### 5) CORS 默认 `permissive`
- **File:** `crates/veda-server/src/main.rs:97`
- **What:** `CorsLayer::permissive()` → `Access-Control-Allow-Origin: *`。
- **Why it matters:** 任意浏览器 origin 可以发带着用户 cookie/bearer 的请求。虽然 bearer token 不会被 browser 自动附加，但如果之后集成 cookie auth 或 SSO，会立刻成为 CSRF / 数据外泄入口。权限式 API 的默认不应是 `permissive`。
- **How to apply:** 从 config 读 `allowed_origins`，默认空（同源）；开发模式显式打开。

### 6) JWT 签发/校验没有算法锁定、无 issuer/aud，也不校验密钥强度
- **File:** `crates/veda-server/src/auth.rs:22-51`、`main.rs:74`
- **What:** `Validation::default()` 包含 exp 检查但 alg 默认是 `HS256`，没有显式 `set_required_spec_claims` 或 `set_audience` / `set_issuer`。配置里允许 `jwt_secret = "change-me"`。
- **Why it matters:** 如果将来切换到 RS/ES 或者把 secret 公共化，默认 Validation 不能防止算法替换攻击（老版本 jsonwebtoken 有过 alg=none bypass）。弱 secret 直接离线爆破即签 token。
- **How to apply:** 启动时校验 `jwt_secret.len() >= 32` 且不为默认值；`Validation::new(Algorithm::HS256)` 显式指定；至少加 `iss = "veda"` 并校验。

### 7) Outbox `fail` 用 `format!` 拼 SQL（backoff 秒数）
- **File:** `crates/veda-store/src/mysql.rs:1369-1383`
- **What:** `let sql = format!("... INTERVAL {} SECOND ...", backoff_secs);`。
- **Why it matters:** `backoff_secs` 源头是服务端计算的 `i64`，目前无注入风险。但这个模式一旦有人拷贝改参数就容易 SQL 注入。
- **How to apply:** 改成 `DATE_ADD(UTC_TIMESTAMP(), INTERVAL ? SECOND)` 并 `.bind(backoff_secs)`。

### 8) FUSE 写句柄 `open` 读全文件进内存
- **File:** `crates/veda-fuse/src/fs.rs:314-343`
- **What:** 任何 writable open 都 `client.read_file(path)` 把整个文件塞进 `WriteHandle.buf`。
- **Why it matters:** O_APPEND 场景完全不需要原内容；1 字节修改的场景不需要 50 MB 内存。多用户并发 `vim large.log` 直接 OOM。
- **How to apply:** 引入"lazy COW buffer"：open 只记录 base_rev，不 read；`read` 缺页时再按 range 读；`write` 累积 delta；`flush` 时根据是否有 `offset == base_size` 的纯 append 选择 `/v1/fs/{path}` POST append vs PUT full rewrite。短期优化：至少 O_APPEND 分支跳过 read_file。

### 9) FUSE `InodeTable` 无上限，长跑会持续泄露
- **File:** `crates/veda-fuse/src/inode.rs:8-14`
- **What:** `ino_to_path` / `path_to_ino` / `attr_cache` 只加不清（`remove_path` 只在 unlink/rmdir 显式调用；普通 lookup/readdir 只加）。
- **Why it matters:** 长跑 daemon 里 `ls -R /big-project` 一次加 10 万项，常驻内存；第二次再 ls 不同项目再加 10 万项。几天就 OOM。FUSE `forget` 回调可以用来驱逐，但当前没实现。
- **How to apply:** 实现 `Filesystem::forget` 驱逐对应 inode；或者 cache 大小超过阈值时清空 cache（`attr_cache` 可以直接丢，ino 映射丢失会导致现有 fh 失效，需要更谨慎的策略）。

### 10) `list_dir_recursive` 是"应用层递归 + 每层一次 SQL"
- **File:** `crates/veda-core/src/service/fs.rs:546-584`
- **What:** 栈式 DFS，每推出一个目录就 `list_dentries(parent_path = dir)` 一次 SQL。
- **Why it matters:** 深度 100 / 宽度 100 的树 = 10 000 次 round trip。已经存在 `list_dentries_under` (单条 LIKE query) 的实现——`glob_files` 用的就是它。`list_dir_recursive` 没改过来。
- **How to apply:** 直接调 `self.meta.list_dentries_under(workspace_id, norm)`，应用层不再递归。（同步解决后，Critical 3 的 LIKE 索引修复也让这条变得高效。）

## Redundancy & Waste

- `MetadataTx::get_dentry` 默认使用 `FOR UPDATE` (`mysql.rs:905`)。但许多调用点只读不写（比如 `write_file`'s fast-path precheck），被迫拿写锁。拆成 `get_dentry` / `get_dentry_for_update` 两个方法会显著减少锁冲突。
- `parse_if_match` (`routes/fs.rs:106-110`) 把"格式非法"和"缺失"视为等同，都返回 `None`——客户端传了错误 If-Match 会静默降级为"无前置条件"。应当 4xx。
- `FilesTable::scan` 在有 filter 时禁用 limit pushdown (`files_table.rs:235-239`)。对 `SELECT * FROM files WHERE path LIKE '/data/%' LIMIT 10` 这类极常见查询仍全拉再截断。filter 是 `Inexact`，保留全量读是安全的，但 LIMIT 只决定"最多吐出多少行"，没有理由因 filter 存在而禁用——可以把 limit 作为"上限"传下去，filter 后不足再多读一批。
- `MysqlStore::get_dentry_path_by_file_id` 对同一个 `file_id` 可能有 >1 条 dentry（copy 后 ref_count > 1），但 `LIMIT 1` 返回的是不确定的哪一条，会让搜索结果显示"随机某个路径"。要么定义"规范路径"，要么让 trait 返回 `Vec<String>`。
- `compute_write_meta_blocking` 的 inline 分支（`fs.rs:69-81`）算完 sha 又单独调 `content.lines().count()` 做另一遍扫。合并成一次遍历省一半 CPU（256 KB 下缓存热，影响小，可做可不做）。
- `crates/veda-sql/src/fs_events_table.rs:149-153` 有 `if n > 0 { MemTable } else { MemTable empty }` 分叉，两条分支语义等价（空 batch 会构造出空表），可以直接用 `batch` 分支一条路。

## Documentation Gaps

- `ARCHITECTURE.md` 标称 Phase 1-6 已完成且 "Worker 启动 + graceful shutdown"；但实际 Worker 轮询依赖静态间隔、SSE 存在 N 连接 × 1 QPS 轮询模式，这些"当前实现的性能特性"没有在文档中说明，容易让用户以为能线性扩展。
- `docs/PLANS.md` 的 "Reconciler" 和 "Prometheus metrics" 仍挂在"待实现"，但两者之间有个没被记录的依赖：没有 metrics 无法评估 reconciler 的 lag，建议在 PLANS 里加一条"先做 metrics 再做 reconciler"。
- 没有一个文档写明"CORS / JWT / API key"这三层的威胁模型，Strategic 5/6 的风险无处可查。建议新增 `docs/security.md` 记录当前安全假设（"假设服务端部署在可信网络内"等）以及生产上需要补齐的项。

## What's Good

- `path::normalize` 对 `..` 越界、绝对路径、NFC 归一化都做了，比很多项目踏实。
- Outbox + Worker 的语义正确：`upsert` 先写新、后删旧，`chunk_sync skipped: file no longer exists` 时主动清理向量存储。
- `MetadataTx` 的 `commit(self: Box<Self>)` 采用 typestate 思路避免了"commit 后仍被使用"的类误用。
- `FilesTable::scan` 做了路径前缀下推（从 `WHERE path = 'x'` / `WHERE path LIKE '/prefix/%'` 推出扫描根）、并用 `needs_file_records` 判断是否批量加载 FileRecord——一旦 Critical 3 的索引修好，这一路径就能立刻受益。
- `write_file` 的 `If-None-Match sha256` 短路在有客户端支持时能完全绕过 hash + DB 写，设计对 chatty append/rewrite 非常友好。

### Key Decisions Observed

- **事务驱动一致性**：所有 fs 修改走 `MetadataTx` + `FOR UPDATE`，outbox 保证跨库（MySQL↔Milvus）最终一致。优点：简单、正确性易推理。代价：锁竞争直接翻译为并发度，路径 6/8 的问题都是这个设计的衍生。
- **Inline ≤ 256 KB / Chunked 其余**：单阈值策略，无中间层。`file_chunks.content MEDIUMTEXT` 上限 16 MB，与 chunk_size 256 KB 有 64× 的余量。
- **SQL 引擎层面做安全网**：`MAX_SCAN_ENTRIES = 100_000`、`MAX_SINGLE_FILE_BYTES`、`MAX_GLOB_TOTAL_READ = 100 MB`——把 "DataFusion 是否会 OOM" 的问题硬挡在入口，做法对。
- **FUSE 作为 thin client**：所有元数据和内容都走 HTTP；本地只做 inode 映射和读缓存。不处理 offline / network partition，这是有意取舍（ARCHITECTURE.md 隐含）。
- **workspace = 强隔离边界**：所有查询都带 `workspace_id`，Milvus filter 也一样；没有跨 workspace 的能力。这让多租户审计容易，但也意味着 SSE 之类的资源无法在 workspace 间共享。
