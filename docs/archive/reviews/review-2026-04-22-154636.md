## Verdict

代码整体质量中上，模块边界基本清晰，但存在多处“同语义多实现”的重复，已经开始抬高维护成本。当前可安全推进一轮“去重型重构”（不改协议/不改存储模型），优先清理 `fs` 写入路径、MySQL pool/tx 双实现、SQL 文件读取护栏重复逻辑。

## Critical Findings

### 1) `FsService` 写入路径重复实现，COW 与事件逻辑分散
- **File:** `crates/veda-core/src/service/fs.rs`
- **What:** `write_file_once` 内含完整 COW/更新/事件流程，同时 `finalize_full_rewrite` 再实现一套近似逻辑。
- **Why it matters:** 一处修 bug、另一处漏改的概率高，尤其在 revision、outbox、fs_event 三者联动时。
- **Recommendation:** 抽取统一私有 helper（例如 `apply_rewrite(meta, old_file, dentry, mode)`），让 `write_file_once` 与 `append_file` 复用同一提交流水线。

### 2) MySQL pool 与 tx 分支存在大段重复 SQL 逻辑
- **File:** `crates/veda-store/src/mysql.rs`
- **What:** `list_dentries_under` 与 `get_file_chunks` 在 `MetadataStore` 和 `MetadataTx` 各写一份；`insert_outbox_conn` 与 `insert_fs_event` 都有 `id==0 / 显式id` 双分支模板代码。
- **Why it matters:** SQL 逻辑分叉会导致行为漂移（尤其是 bind 顺序、范围条件语义）。
- **Recommendation:** 提取共享 query builder/helper（仅执行器差异化），先收敛 `list_dentries_under`，再收敛 `get_file_chunks`。

### 3) SQL 文件读取护栏逻辑重复（UDF 与 Table Function）
- **File:** `crates/veda-sql/src/fs_udf.rs`, `crates/veda-sql/src/fs_table.rs`
- **What:** `veda_read` 与 `read_single_file` 都在做 `stat -> MAX_SINGLE_FILE_BYTES 检查 -> read_file`。
- **Why it matters:** 限制条件、错误消息、NotFound 语义易分叉。
- **Recommendation:** 合并为单一 bounded-read helper，两个入口复用，统一错误映射。

## Strategic Concerns

### 1) 搜索模型分裂：`SearchRequest` + `HybridSearchRequest` + `VectorStore` 双方法
- **File:** `crates/veda-types/src/types.rs`, `crates/veda-core/src/store.rs`, `crates/veda-core/src/service/search.rs`
- **What:** 同一“搜索”概念拆成两个 request 和两个 trait 方法。
- **Why it matters:** 服务层和后端实现都要做分支拼装，扩展 mode 时修改面大。
- **Recommendation:** 收敛为单 request + 单 `VectorStore::search`，用字段约束表达语义（例如 `query_vector/query_text`）。

### 2) API DTO 重复映射：`SearchHit` -> `SearchResultItem`
- **File:** `crates/veda-types/src/types.rs`, `crates/veda-types/src/api.rs`, `crates/veda-server/src/routes/search.rs`
- **What:** 路由层手工搬运字段并做 `path.unwrap_or_default()`。
- **Why it matters:** DTO 变更时容易忘改映射，默认值策略散落。
- **Recommendation:** 增加 `From<SearchHit> for SearchResultItem` 或合并为单一传输模型。

### 3) CLI 与 FUSE 的 HTTP 客户端语义重复且有轻微行为差异
- **File:** `crates/veda-cli/src/client.rs`, `crates/veda-fuse/src/client.rs`
- **What:** URL 拼接、`trim_start_matches('/')`、状态码处理、哈希 header 逻辑重复实现。
- **Why it matters:** 新接口或 header 约定会在两个客户端分头维护。
- **Recommendation:** 提取轻量 shared client-core（至少共享 URL 规则与错误映射），async/blocking 分层保留。

### 4) 路由层权限检查模板重复
- **File:** `crates/veda-server/src/routes/fs.rs`, `crates/veda-server/src/routes/collection.rs`, `crates/veda-server/src/routes/account.rs`
- **What:** 多处重复 `auth.read_only` 拒写、workspace owner 校验。
- **Why it matters:** 后续新增路由时容易漏安全校验。
- **Recommendation:** 抽 `require_write()` 与 `load_owned_workspace()` helper，统一失败语义。

## Redundancy & Waste

- `crates/veda-fuse/src/fs.rs` 的 `parent_path` 与 `crates/veda-fuse/src/sse.rs` 的 `parent_dir` 重复。
- `crates/veda-sql/src/search_table.rs` 和 `crates/veda-sql/src/fs_events_table.rs` 重复 `if n > 0 { MemTable } else { empty MemTable }`。
- `crates/veda-types/Cargo.toml` 声明了 `uuid`，但 `veda-types` 源码无使用。
- `crates/veda-sql/tests/sql_test.rs` 并存 `MockMetaFull` 与 `MockMeta`，大量 trait 实现重复，测试维护成本偏高。

## Documentation Gaps

- `docs/design.md` 仍以 WebSocket 事件为主叙述，而运行实现已包含 `GET /v1/events` SSE 路径，建议补一条“为何 SSE/WS 并存或替换”的决策记录。
- `docs/PLANS.md` 的 Phase 4 项仍显示 WebSocket endpoint 未完成，和当前代码能力存在偏差，建议更新状态与验收口径。

## What's Good

- crate 拆分总体合理，`veda-core` 的 trait 边界（元数据、向量、认证、集合）清楚。
- `FsService` 关键路径已有较强测试覆盖，给去重重构提供了安全网。
- `veda-server` 作为薄壳架构方向是对的，问题主要是重复模板而非架构错误。

### Key Decisions Observed

- 以“最终一致 + outbox”解决 MySQL/Milvus 双写，方向正确；当前问题主要是实现层重复，不是策略错误。
- SQL 层通过 UDF + UDTF 扩展能力，功能完整但参数解析/空结果处理存在可收敛样板。
- 客户端侧采取 CLI async + FUSE blocking 双栈，工程上合理，但需要 shared core 防止协议细节漂移。
