# Veda 项目深度 Review 报告

**日期:** 2026-04-17
**Reviewer:** Claude (deep-review)
**范围:** 项目目前为止的全量实现（跨 7 个 crate，包含未提交改动）
**提交分析数:** 最近 5 个已提交 commit + 6 个文件的未提交改动
**文件分析数:** 约 30 个核心源文件

## Verdict

整体架构是**清晰且成熟**的：MySQL 控制面 + Milvus 数据面 + Outbox 的分工、内容寻址 dedup、COW + 引用计数、分层存储（inline/chunked），这些关键决定都下得很准。代码也有 TDD 痕迹，错误类型和事务边界处理在大多数地方是到位的。

但当前代码 **尚未到可上生产的状态**：有若干会在规模或并发下被触发的 correctness 问题（最严重的是 `upsert_chunks` 非原子 delete→insert、`list_dentries` 可能无索引全表扫描、`append_file` 事务中的重复查询），以及一批 DataFusion 查询路径上的性能塌方点（无 `filter/limit` 下推、per-row `block_on`）。在把规模拉到 10 万文件以上或进入多 worker 并发处理之前，建议先把本报告的 Critical 全部修掉。

Strategic 层面 3 个值得尽早决策的事：**Milvus 写入一致性**、**Worker 并发模型**、**DataFusion 与 `MetadataStore` 的查询契约（filter 下推接口）**。

## Critical Findings

### 1. `upsert_chunks` 非原子：删除已成功、插入失败时向量库永久少一批 chunks

- **File:** `crates/veda-store/src/milvus.rs:369-413`
- **What:** `upsert_chunks` 实现是「先 delete 掉该 file_id 所有旧 chunks，再 batch insert 新 chunks」。Milvus REST 的两个 HTTP 调用之间没有事务。如果进程被 kill / 网络抖动 / 第二步失败，Milvus 中这个 `file_id` 就永久丢向量了——而 MySQL 里的 file record 仍是正常 `revision`，worker 完成任务前还会 `task_queue.complete(task.id)`，没有自愈机制。
- **Why it matters:** 这是**数据悄悄丢失**，不是可观测的错误。搜索命中率会缓慢下降，没人会发现。
- **Recommendation:**
  - 正确顺序应该是 **insert-then-delete-old**（按 chunk_index 插入新 id，再删除 `chunk_index >= new_count OR revision != new_rev` 的旧行），中间即便失败也只是短暂有重复，不会丢数据。
  - 或者在 `ChunkWithEmbedding.id` 里带上 revision，`delete_chunks` 按 `revision < new_revision` 删除，插入永远追加。
  - 同时 worker 侧改成「成功后再 complete」，失败时不 complete，让 `fail()` 走重试。

### 2. `list_dentries` 极大概率全表扫描（缺 `(workspace_id, parent_path)` 或 `(workspace_id, path)` 复合索引）

- **File:** `crates/veda-store/src/mysql.rs:371`（schema DDL 附近）、`list_dentries` 查询 `WHERE workspace_id = ? AND parent_path = ?`；以及 `get_dentry` 用 `WHERE workspace_id = ? AND path = ?`
- **What:** 目前表上只有 `(workspace_id, path_hash)` 唯一索引 + `(workspace_id, parent_path(255))` 前缀索引。两个问题：
  1. `get_dentry` 用 `path` 查询，不是 `path_hash`，而 `(workspace_id, path)` 没有直接索引，查询优化器不一定能走 `path_hash` 索引（MySQL 不会自动做 hash(path)）。
  2. `parent_path(255)` 前缀索引对深路径（>255 字节）会出现索引前缀冲突，即便能走索引也会有大量 false positive 回表。
- **Why it matters:** `list_dentries` 每次 `ls` 都在调用；`FilesTable::scan` 的递归扫描会以 O(目录数) 次调用它。工作区文件数达到万级时，每个 SELECT 用时会从 ms 涨到秒级。
- **Recommendation:**
  - 查询侧统一用 `path_hash`：`WHERE workspace_id = ? AND path_hash = ? AND path = ?`（`path_hash` 保证走索引，`path` 兜底防 hash 冲突）。
  - 扫描用：新增 `INDEX idx_parent (workspace_id, parent_path_hash)` 或者直接升级到 `VARCHAR(768)` 全列索引（utf8mb4 下 MySQL 8 已支持）。
  - 把决定落文档，这是数据库级 schema 决定，不应默默生效。

### 3. `append_file` 在同一事务内对同一 file_id 调用 `tx.get_file()` 两次

- **File:** `crates/veda-core/src/service/fs.rs:638` 和 `crates/veda-core/src/service/fs.rs:672`
- **What:** 第一次（638 行）读出来为了拼接内容和校验 append 配额；第二次（672 行）读出来只是为了再用一次 `f.revision` / `f.checksum_sha256` / 判定 `content_unchanged`。这两份数据在同一事务里必然一致，没有必要二次查询。
- **Why it matters:** 小文件 append 的路径被放大 50%—100% 的 DB RTT；`append_file` 已经是热点路径（日志/追加写）。
- **Recommendation:** 在 638 行就把 `file` 绑定成外层 `let`，后面直接复用，删掉 672 行。

### 4. `veda_fs_events` 表函数 `limit` 参数未做符号校验

- **File:** `crates/veda-sql/src/fs_events_table.rs:78`
- **What:** `limit = *v as usize`，`v` 是 `i64`。SQL 里传 `-1` 会被强转成 `usize::MAX - 0`（实际接近 `18446744073709551615`），传给 `query_fs_events` 会直接把 MySQL `LIMIT` 撑成极大值。
- **Why it matters:** 一次恶意/误用 SQL 即可内存打爆（整张 `fs_events` 加载）。
- **Recommendation:** `if *v < 0 { return Err(...) } let limit = (*v as usize).min(MAX_FS_EVENTS_LIMIT);`，同时为 `query_fs_events` 定一个 hard cap（例如 100k）。

### 5. `FilesTable::scan` 完全忽略 `filters` 和 `limit`，每次都把工作区全量 dentry + file 拉进内存

- **File:** `crates/veda-sql/src/files_table.rs:61-108`
- **What:** `_filters` 和 `_limit` 都是 `_` 开头被丢弃的。无论 `SELECT * FROM files WHERE path = '/a.md' LIMIT 1` 还是 `SELECT count(*) FROM files`，实现都做「广度优先列出整个文件树 → batch 取所有 FileRecord → 全量 Arrow builder」。
- **Why it matters:** 一个工作区到 10k 文件时，每个简单 SQL 都要扫全量 + 在 Rust 侧聚合。DataFusion 做不了 pushdown，因为你没实现 `supports_filters_pushdown`。
- **Recommendation:**
  - 最小修：实现 `supports_filters_pushdown`，把 `path = '…'`、`path LIKE '/foo/%'`、`is_dir = true/false` 下推到 `MetadataStore`。
  - 把 `limit` 传下去。
  - 长远：给 `MetadataStore` 加一个 `scan_dentries(workspace_id, filter, limit)` 接口，让 SQL 层和 FS 层共享同一个索引查询。

### 6. Worker `handle_chunk_sync` 对 chunked 文件做全量内存 reassemble

- **File:** `crates/veda-server/src/worker.rs:112-116`
- **What:** `StorageType::Chunked` 分支 `chunks.into_iter().map(|c| c.content).collect()` 是一次性把所有 chunks 拼成一个 `String`，然后再 `semantic_chunk(&content, 2048)` 重新切。对于 `INLINE_THRESHOLD=256KB` 以上、可能几 MB 甚至几十 MB 的文件，这是一次 **完整拷贝**。
- **Why it matters:** worker 的 batch_size 是配置项；batch 里若有 3-5 个大文件，RSS 立刻翻倍。
- **Recommendation:**
  - 短期：在 `chunks` 层面流式处理——`semantic_chunk` 要么改成支持 `Iterator<Item=&str>` 输入，要么保留当前 API 但在 `worker` 里按 chunk 滑动窗口。
  - 中期：chunk 大小已经是 2048 token，`veda-pipeline::chunking::semantic_chunk` 对已有的 `FileChunk` 可以复用其边界，不必完全重切。
  - 退一步至少：`String::with_capacity(total_size)` 避免 realloc。

### 7. `fs_udf` 中 scalar UDF 的每一行都会 `block_on` 一次

- **File:** `crates/veda-sql/src/fs_udf.rs:55-67`（`block_on`）以及各个 invoke 分支（按每行取值再调 `fs_service.read_file(...).block_on()`）
- **What:** `veda_read('/foo.md')` 对 1 行是对的，但 `SELECT veda_read(path) FROM files` 时，`get_string_values` 返回 `Vec<String>`（大小 = `num_rows`），下面的实现（具体见 `invoke_with_args`）会对每一行 spawn 一个 `block_in_place` + `block_on`。
- **Why it matters:** `block_in_place` 会让当前 worker 线程退出调度、请调度器补一个新线程；N 行 N 次切换对小行数可忍，10k 行就会把 runtime 线程池打穿。
- **Recommendation:**
  - 给 UDF 加快路径：如果 input 是 `ColumnarValue::Scalar`，只 block_on 一次。
  - 数组路径用 `handle.block_on(futures::future::try_join_all(...))` 合并。
  - 同时 `veda_write`/`veda_append` UDF 没有 size 上限，用户可以 `veda_write(path, very_long_text)`，会绕过 `MAX_APPEND_FILE_BYTES`（50MB）。把 `MAX_WRITE_FILE_BYTES` 也加上，整个 fs 层共用一个常量。

### 8. `append_file` 50MB 上限与 `write_file` 无限制不对称

- **File:** `crates/veda-core/src/service/fs.rs:641`（append 有 MAX_APPEND_FILE_BYTES 检查）、同文件 `write_file` 无等价检查
- **What:** `append_file` 对 `file.size_bytes + content.len()` 做了 50MB 限制；`write_file` 没有任何 size 限制。一个 100MB 的 POST body 会直接被 inline/chunked 写进去（chunked 分支会 OK，但内存中仍需完整持有原 content）。
- **Why it matters:** 一致性 + 内存/磁盘防御。对调用方来讲，"write 没限制 append 有"是反直觉的。
- **Recommendation:** 抽 `MAX_FILE_BYTES` 作为全局常量（例如 256MB），`write_file` / `append_file` / UDF `veda_write` 共用。

### 9. `login` 每次调用都生成一把新 API Key（无论是否已有）

- **File:** `crates/veda-server/src/routes/account.rs:100-111`（`login` handler）
- **What:** 账号密码校验通过后，handler 直接 `create_api_key`，返回原 key。没有判定「该账号是否已有活跃 key」、也没有做 rotate 语义。
- **Why it matters:** 长期运行下 `api_keys` 表会持续膨胀，而且没有清理机制；前端每次「重新登录」就留下一个孤儿 key。其次这也削弱了 key 泄露检测能力——没有"当前活跃 session"的概念。
- **Recommendation:**
  - 短期：`login` 成功后返回 JWT（而不是 API key）。API key 是长期凭证，应当 explicit 创建（走 `create_api_key` 路由）。
  - 如果坚持当前返回 key 的行为，至少加 `WHERE account_id = ? AND last_used_at > NOW() - INTERVAL 7 DAY` 的复用逻辑。

### 10. `delete_workspace` 只归档，遗留数据无 GC

- **File:** `crates/veda-server/src/routes/account.rs`（`delete_workspace` handler）
- **What:** 只做了 `workspace` 表 soft delete，dentries / files / file_contents / file_chunks / outbox / fs_events / Milvus 集合都没有清理。
- **Why it matters:** 删除 workspace 后存储不回收；Milvus 依然保留向量；chunk_sync outbox 任务仍在跑，worker 会操作不存在的 workspace。存储和预算会漂移。
- **Recommendation:**
  - 定义清楚「soft delete + 异步 GC」还是「级联硬删」。目前偏向前者，那就要加一个定时 GC worker 任务（类似 outbox）。
  - 最小止血：`delete_workspace` 发一个 `WorkspaceDelete` outbox 事件，由 worker 串行处理 fs + Milvus 清理。

## Strategic Concerns

### S1. `MetadataStore` trait 没有 filter/limit 语义，注定 DataFusion 查询会重复痛

- **File:** `crates/veda-core/src/store.rs:7-47`
- **What:** `list_dentries(workspace_id, parent_path)` 是唯一的批量 API，无 filter、无 limit、无分页。SQL 层想做 pushdown 时没有接口可用，只能要么全拉要么改 trait。
- **Why it matters:** 当前 `FilesTable`、未来 `FsEventsTable` 的 pagination、所有"按前缀/按类型过滤"的场景都会被这个 trait 卡住。
- **Recommendation:** 趁 trait 还没被大量实现，加一个 `ListOptions { path_prefix, is_dir, limit, offset, cursor }` 参数；sqlite mock 可以忽略选项做全扫（反正数据量小），MySQL 实现下推。

### S2. Worker 串行处理 + 固定 `poll_interval` = 吞吐量天花板很低

- **File:** `crates/veda-server/src/worker.rs:53-71`
- **What:** `poll_once` 一批 claim 到的 task 是 `for task in tasks` 串行处理。单个 chunk_sync（1 MB 文件）要走 OpenAI embedding（几百 ms）+ Milvus insert。N 个任务就是 N × 几百 ms，即使底层有并发能力。
- **Why it matters:** 用户 upload 一个 1000 文件的工作区，现在的 worker 要跑几分钟。
- **Recommendation:**
  - `futures::stream::iter(tasks).buffer_unordered(N).collect()` 加并发。
  - `embedding.embed(&texts)` 已经是 batch 接口——可以跨任务合并 embedding 调用（需要一个小 batcher）。
  - poll interval 改成 exponential backoff + notification（MySQL 里 INSERT outbox 时 `NOTIFY` 不可用，可以用 tokio watch channel 从 service 侧 poke worker）。

### S3. Milvus 混合搜索静默 fallback 到 ANN

- **File:** `crates/veda-store/src/milvus.rs`（`hybrid_search_remote` 分支）
- **What:** 如果 Milvus 版本不支持 BM25 或 hybrid 搜索，代码会 silently fallback 到纯向量 ANN，没有 log 级别提升、没有响应头或元数据透出。
- **Why it matters:** 用户以为自己在做 hybrid 搜索，实际只有 dense 向量。调试时极难复现。
- **Recommendation:** fallback 时 `warn!` 一次 + 在响应里带一个 `search_mode: "vector_only" | "hybrid"` 字段。

### S4. JWT 硬编码 24h TTL + `read_only: false`

- **File:** `crates/veda-server/src/auth.rs`（`create_jwt`）
- **What:** TTL 写死在代码里；JWT claim 里 `read_only` 永远是 false（`workspace_key` 的 read_only 属性根本没被写进 JWT）。
- **Why it matters:**
  - 合规 / 运维侧的会话时长是个 tunable，不应硬编码。
  - read_only workspace key 在签发 JWT 后就失去 read-only 属性，任何 JWT 都能写——这**是个权限 bug**。
- **Recommendation:**
  - `Config` 里加 `jwt_ttl_seconds`。
  - `create_jwt` 接收 `read_only: bool`，写入 claim；`fs_udf` 已经有 `check_write_allowed`，扩展到 HTTP 路由层。

### S5. MySQL 的 `claim()` 是 SELECT→N 次 UPDATE

- **File:** `crates/veda-store/src/mysql.rs`（TaskQueue::claim）
- **What:** `SELECT ... FOR UPDATE SKIP LOCKED` 拿到 N 个 id 后，循环 `UPDATE` 每一条。这是 N+1 round trip。
- **Why it matters:** 当 batch_size=50 时每次 poll 51 次 RTT。不是阻塞式致命，但浪费明显。
- **Recommendation:** 改成 `UPDATE ... WHERE id IN (?, ?, ...)` 单次更新，或直接用 `UPDATE ... WHERE status='pending' ORDER BY id LIMIT N` + `RETURNING`（MySQL 8 支持）跳过 SELECT。

## Redundancy & Waste

### R1. `mkdir` 用错误消息 `contains("Duplicate entry")` 判定目录已存在

- **File:** `crates/veda-core/src/service/fs.rs:505` 附近
- **What:** 依赖 MySQL 错误 message 字符串做分支。sqlx 升级 / MariaDB 兼容性 / 未来换 Postgres，这个判断会静默失效。
- **Recommendation:** 先 `tx.get_dentry()` 检查是否存在，或者通过 sqlx 的 `Error::Database(db_err)` 提取 `code()`（MySQL 1062），不要用消息字符串。

### R2. `FilesExec::new` 重建 `cache` 用的 `projected_schema` 可能和实际 project 不一致

- **File:** `crates/veda-sql/src/files_table.rs:210-217`
- **What:** `execute` 里对 `batch.project(&indices)`，`indices` 来自 `projected_schema.fields().iter().filter_map(|f| self.full_schema.index_of(f.name()))`。如果 `project_schema` 重命名字段，这个 `filter_map` 会静默丢列。
- **Why it matters:** 不是 live bug（现在 projection 不重命名），但脆弱。
- **Recommendation:** 直接用 DataFusion 给的 `projection: Option<&Vec<usize>>`，在 `FilesExec` 里存起来用——schema 自己查 DataFusion 不要重推。

### R3. `get_files_batch` trait default impl 是纯 N+1

- **File:** `crates/veda-core/src/store.rs:12-20`
- **What:** default impl 循环 `get_file(id)`。mock 能接受，但真实后端如果忘了 override 就会性能塌方。
- **Recommendation:** 留 default 但日志 `warn_once!("using N+1 default impl of get_files_batch ...")`，或者改为 `unimplemented!()` 逼实现方显式选一边。

### R4. Outbox `fail()` 线性 retry 30s

- **File:** `crates/veda-store/src/mysql.rs`（`fail` 方法）
- **What:** 失败任务被标 `next_retry_at = NOW() + 30s`，不管第几次失败都是 30s。
- **Why it matters:** 永久挂掉的任务永远在 worker 重试清单里循环。
- **Recommendation:** `next_retry_at = NOW() + min(30 * 2^retry_count, MAX_BACKOFF)`，同时 `retry_count > N` 进死信。

### R5. `post_v2` 对 response 里的 `code` 字段 `or_else` 两次

- **File:** `crates/veda-store/src/milvus.rs`（`post_v2` 分支）
- **What:** 先尝试 `as_i64`，再 `as_u64 as i64`，逻辑重复。不是 bug，读起来费劲。
- **Recommendation:** `.as_i64().or_else(|| obj.get("code").and_then(|v| v.as_u64().map(|n| n as i64)))`，一行。

## What's Good

- **Outbox 模式选对了**：控制面事务 + 异步写向量库，是这类系统正确的一致性选择。`insert_outbox` 和 fs 操作同事务，worker 幂等消费，整套链路心智负担很小。
- **COW + ref_count 逻辑正确**：`fs.rs:60-105` 的 `write_file` 走 dedup 命中路径时，`increment_ref_count` 在事务内、`decrement_ref_count` 在 delete 路径、重命名不动 ref_count——语义是对的。
- **`path_hash` 设计**：虽然现在查询路径没完全用上（见 Critical #2），但 `path_hash` 列的存在给了后续修复一个简单口子，不需要迁移 schema。
- **分层存储阈值 256KB**：对 LLM 场景（典型文档 1-100KB）这个阈值命中率高，inline 路径省了一次 chunk 表查询。选得很准。
- **`FieldDefinition` serde rename 修复**（未提交 diff）：`#[serde(rename = "type", alias = "field_type")]` 同时兼容了 Milvus 协议和内部旧字段名，干净。
- **DataFusion 的使用克制**：没为了"体现技术"到处塞 UDF，`FilesTable` / `FsEventsTable` / `fs_udf` 各自边界清晰。

### Key Decisions Observed

1. **MySQL 做控制面 + Milvus 做数据面**：控制面选 MySQL 而不是 Postgres，估计是 Joe 对 MySQL 更熟 + 运维生态成熟。代价是 `RETURNING` / `NOTIFY` / 数组类型 / JSON 索引这些 Postgres 强项都用不上。现在还没触底，但 outbox polling 和 `claim()` 的 N+1 都是 Postgres 下会更优雅的场景。
2. **Outbox + Worker 单机**：目前 worker 是进程内 task，不是独立服务。好处是部署简单，坏处是 API 负载和处理负载共享线程池，重 embedding 会影响 API 响应。
3. **内容寻址 dedup 只在 workspace 内做**：`find_file_by_checksum` 带 workspace_id。跨 workspace 不共享存储——这是正确的 tenant 隔离决定，但如果将来出现 org 级共享资源，会需要重新思考。
4. **Inline 阈值 256KB**：偏保守的选择（很多系统用 1MB+）。优势是 MySQL 行不会太肥，`get_file_content` 走主表即可；代价是 chunked 路径被更频繁激活。
5. **JWT + API Key 双凭证**：API key 长期、JWT 短期。这是标准选择，但目前 JWT 没带 read_only 属性（见 S4），未来要补权限传递链。
6. **DataFusion 作为可选 SQL 前端而非主查询引擎**：`FilesTable` 的实现方式（忽略 filter/limit）说明 Joe 把它定位为"偶尔用用的 read-only 视图"，不是 OLTP 入口。这个定位是对的，但代码里没写清楚，未来容易被误用。
