## Verdict

Sprint 3 is **close to ship-ready with one real correctness concern** (C1 — `append_file` non-atomic + full-rewrite cost) and a small set of polish items. The good news: **Sprint 1's Critical #1 (read-only gate) is now fully wired** — `VedaSqlEngine::execute` takes `read_only: bool` and `routes/sql.rs:23` passes `auth.read_only`, closing the security gap from the last two reviews. The new UDTFs are small, well-bounded (sensible `DEFAULT_LIMIT`, proper `escape_like` for SQL injection), and tested. But `append_file` is a pure read-modify-write with no transaction — for the log-append use case the plan targets, this is both a concurrency hazard and a hidden performance cliff (every 100-byte append to a 40MB log re-reads, re-SHA256s, re-writes, and re-embeds the whole file). That should be the only blocker for merge.

## Critical Findings

### C1. `append_file` is non-atomic AND pays full-file cost per call
- **File:** `crates/veda-core/src/service/fs.rs:620-648`
- **What:**
  ```rust
  pub async fn append_file(&self, workspace_id, raw_path, content) -> ... {
      let existing = self.meta.get_dentry(workspace_id, &norm).await?;  // no tx
      let old_content = Some(self.read_file(workspace_id, &norm).await?);  // no tx
      let new_content = old + content;
      self.write_file(workspace_id, &norm, &new_content).await  // starts its own tx
  }
  ```
  Three independent DB calls, no transaction spanning read→write. Plus: `read_file` loads the entire file, we allocate `old + content`, then `write_file` writes it all back, SHA256s the whole thing, and emits a `ChunkSync` outbox that triggers full-file re-embedding.
- **Why it matters:**
  1. **Lost writes under concurrency.** Two `veda_append('/logs/app.log', 'line\n')` calls racing: both read the same snapshot, both write back. One append's content vanishes. This is exactly the use case the plan targets (`SELECT veda_append('/logs/app.log', ...)`).
  2. **O(file_size) cost per append.** A 40 MB log file + 100-byte append costs ~40 MB read + 40 MB string concat + 40 MB SHA256 + 40 MB write + embedding recomputation. The logging use case inverts: appends are frequent, files grow, cost grows quadratically per append over the file's lifetime.
  3. **No size cap.** `veda_read` has `MAX_READ_SIZE = 50 MB`; `append_file` bypasses it via direct `self.read_file`. A 500 MB file append succeeds (slowly) without any guardrail.
  4. The plan doc acknowledges "并发 append 可能丢失数据" but the risk table entry says "MySQL 事务保证单个 append 原子" — which is *not* true with the current implementation (the read is outside the write's transaction).
- **Recommendation:**
  Short term (before merge): (a) move the whole operation into a single `begin_tx`→`get_dentry`→`read content`→`write content`→`commit` transaction, so two concurrent appends serialize and the second observes the first's write; (b) add a file-size cap equal to `MAX_READ_SIZE` before the read; (c) fix the plan doc to reflect actual atomicity guarantees.
  Medium term: introduce a chunk-append fast path — for `StorageType::Chunked` files, insert a new chunk instead of rewriting. This also avoids the full-file re-embedding, which is the real killer for logs.

## Strategic Concerns

### S1. `veda_fs_events()` silently accepts wrong-typed arguments
- **File:** `crates/veda-sql/src/fs_events_table.rs:49-71`
- **What:** The `if let ScalarValue::Int64(Some(v))` / `ScalarValue::Utf8(Some(v))` pattern falls through silently if the type doesn't match. `veda_fs_events('abc')` — where user clearly meant the first arg to be `since_id` — returns ALL events (default 0). `veda_fs_events(100, 123)` passes `path_prefix=None` and happily runs.
- **Long-term framing:** This is exactly the kind of silent-wrong-default behavior that ships bugs in production queries. Users will stare at their dashboard wondering why their filter isn't working.
- **Recommendation:** On type mismatch or explicit `NULL`, return `plan_err!("veda_fs_events: argument N must be <type>")`. It's 5 lines and catches a class of footguns.

### S2. `event_type` rendered via `format!("{:?}", …).to_lowercase()`
- **File:** `crates/veda-sql/src/fs_events_table.rs:94`
- **What:** Relies on `Debug` output of the `FsEventType` enum, then lowercases. Works today because all variants are single-word (`Create`/`Update`/`Delete`/`Move`), so `{:?}` → `"Create"` → `"create"` matches the serde snake_case.
- **Long-term framing:** The day someone adds `FsEventType::ContentIndexed`, `.to_lowercase()` yields `"contentindexed"` while serde would emit `"content_indexed"` — silent divergence between SQL output and API wire format. A test using Debug-format will happily pass.
- **Recommendation:** Add `impl FsEventType { fn as_str(&self) -> &'static str }` in `veda-types` and use it here and in the MySQL row mapper. One source of truth.

### S3. `storage_stats.total_bytes` counts deduped files once per dentry, not once per file
- **File:** `crates/veda-store/src/mysql.rs:728-748`
- **What:** The query `LEFT JOIN veda_files f ON d.file_id = f.id` joins every dentry to its file. If two dentries share a `file_id` (dedup / copy), `SUM(f.size_bytes)` counts that file's bytes twice.
- **Why it matters:**
  - Is the intent "logical bytes" (what the workspace looks like to the user) or "physical bytes" (what storage actually costs)? They differ whenever dedup fires or users `POST /v1/fs-copy`. Today the implementation returns the former, the plan doc says "总字节数" without clarifying.
  - Users reading "1.2 TB stored" will assume that's their storage bill. It isn't.
- **Recommendation:** Either (a) change the query to `SUM(DISTINCT f.size_bytes)` keyed by `f.id` — actually, just do `SELECT SUM(size_bytes) FROM veda_files WHERE workspace_id = ?` for physical bytes, then join separately for file/dir counts; or (b) keep the current "logical" semantic and add a second column `physical_bytes`. Either way, document it.

### S4. All three UDTFs materialize into MemTable — same eager-I/O pattern as Sprint 2
- **Files:** `fs_events_table.rs:114`, `storage_stats_table.rs:58`
- **What:** Same design as `veda_fs()`: the TableFunction call runs the query at plan time and wraps results in a `MemTable`. For events with `DEFAULT_LIMIT = 1000` this is fine in practice; for stats (1 row) it's trivially fine.
- **Long-term framing:** The pattern locks you into "planning does I/O." When someone writes `SELECT ... FROM veda_fs_events() WHERE id BETWEEN 1M AND 2M LIMIT 10`, the default-1000 limit will bite — the user's LIMIT 10 cannot be honored because the UDTF already applied its own 1000. Predicate pushdown can't reach `query_fs_events`.
- **Recommendation:** No action for this sprint. But the next time a UDTF is added (Parquet? iceberg?), introduce a real `TableProvider::scan` shape and retire the MemTable shortcut. The technical debt is getting cheaper to pay now than later.

### S5. `DEFAULT_LIMIT = 1000` for events is silent and unconfigurable
- **File:** `crates/veda-sql/src/fs_events_table.rs:16`
- **What:** If the user omits the limit argument, 1000 rows is the cap. No warning when the limit hits — query returns exactly 1000 rows, user assumes there are no more.
- **Recommendation:** Either (a) log at `warn!` level when result count hits the limit ("returned 1000 rows, may be more — pass an explicit limit"), or (b) return one extra row-shaped marker in a separate API. At minimum, document that `veda_fs_events()` is cursor-based and silent truncation is expected.

- **File:** `crates/veda-sql/tests/sql_test.rs` (Phase 3 diff in `MockMetaFullTx::insert_fs_event`)

### Key Decisions Observed

- **`append_file = read + concat + write_file`.** A deliberate simplification — reuse `write_file`'s dedup + chunking + outbox path rather than build a parallel code path. The tradeoff (O(file) per append) is real and the log-append use case exposes it. This will be the first refactor target when someone does bulk log ingestion.
- **`event_type` Debug-format shortcut.** Cheap today, fragile tomorrow (S2). Reflects the "ship Phase 3 fast" priority — understandable but worth cleaning up before Phase 4 stabilization.
- **`MemTable` for all UDTFs.** Consistent with Sprint 2 — the whole UDTF subsystem is plan-time-I/O. Stats is fine (1 row), events is fine with a default limit, but the pattern is now load-bearing and will need to change before a streaming use case (Parquet, full-table scans) lands.
- **Positional-only UDTF args.** DataFusion's UDTF doesn't support named args, so `veda_fs_events(0, '/logs/', 100)` is the full surface. The simple string-matching plan doc shows the call pattern `(since_id => 1000, path_prefix => '/logs/')` — DataFusion won't parse that as named args today. Plan doc is slightly aspirational; actual usage is positional. Not a bug, but a doc/impl drift.
- **`POST /v1/fs/{path}` = append, `PUT` = overwrite.** Clean REST verb split. Alternative would have been `?append=true` query param; the verb-based choice is cleaner. `POST` also creates the file if absent, matching `append_file`'s behavior.
