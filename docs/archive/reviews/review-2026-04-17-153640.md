## Verdict

当前实现**不建议直接上线**。核心原因是 `append` 路径存在 COW 语义破坏，会导致“复制后文件被联动修改”的数据正确性问题。其余改动方向（批量加载文件元信息、line_count 溢出保护）是正向的，但在大规模数据下仍有明显可扩展性风险，需要补强后再合入。

## Critical Findings

### 1) `append_file` 破坏 COW 语义，可能导致跨路径内容串改
- **File:** `crates/veda-core/src/service/fs.rs`
- **What:** `write_file` 在 `ref_count > 1` 时会 fork 新 `file_id`（正确的 COW），但 `append_file` 对共享文件直接 `update_file_revision` + 覆写内容，没有对应分叉逻辑。
- **Why it matters:** `copy_file` 后两个路径共享同一 `file_id`，此时对其中一个路径执行 append，会把另一路径的内容一起改掉，违反文件系统语义并造成用户数据污染。
- **Recommendation:** 在 `append_file` 中复用 `write_file` 的 COW 分支（或先统一到一个 `mutate_file_content` 内核函数），并补测试：`copy -> append(one side) -> verify other side unchanged`。

## Strategic Concerns

### 1) `FilesTable` 扫描无谓词下推，固定全量拉取，扩展性风险高
- **File:** `crates/veda-sql/src/files_table.rs`
- **What:** `scan()` 忽略 `_filters` / `_limit`，先递归列出全部 dentry，再批量拉取全部 `FileRecord`，最后交给 DataFusion 上层做筛选。
- **Why it matters:** `SELECT * FROM files WHERE path LIKE '/x/%' LIMIT 10` 在大 workspace 仍是全量 IO + 全量内存构建，P99 会随文件数线性恶化。
- **Recommendation:** 至少支持目录前缀与 limit pushdown；进一步可做流式 RecordBatch（分页拉取）代替一次性内存构建。

### 2) `get_files_batch` 使用单条超长 `IN (...)`，缺少分片保护
- **File:** `crates/veda-store/src/mysql.rs`
- **What:** 按 `file_ids.len()` 动态拼接占位符，未做去重/分批。
- **Why it matters:** 大目录或大量 hardlink 场景下，SQL 长度和 bind 数可能触发 DB 限制（max_allowed_packet / prepared statement 参数限制）或明显增加解析成本。
- **Recommendation:** 先 `HashSet` 去重，再按固定 chunk（例如 500/1000）分批查询并合并结果。

### 3) `CollectionSync` 事件被直接 `complete`，语义与 Outbox 设计不一致
- **File:** `crates/veda-server/src/worker.rs`
- **What:** `CollectionSync` 分支仅打日志后 `complete`。
- **Why it matters:** 未来若真正投递该事件，会出现“任务成功但未执行”的静默丢失，且不进入重试/死信路径。
- **Recommendation:** 未实现前应 `fail`（可带明确错误码）或在入队侧禁止该事件，避免 false-positive completion。

## Redundancy & Waste

### 1) `get_file_chunks` 解码策略不一致，且事务版本存在静默兜底
- **File:** `crates/veda-store/src/mysql.rs`
- **What:** 非事务版本复用 `row_to_file_chunk`（严格错误传播）；事务版本手写映射并 `unwrap_or_default`。
- **Why it matters:** DB 字段异常时事务版本会悄悄产出空内容/0 值，问题被掩盖，后续可能演变为“成功但数据错”的隐蔽故障。
- **Recommendation:** 两处统一复用 `row_to_file_chunk`，严禁在存储层吞解码错误。

## Documentation Gaps

- 设计文档保留了 `CollectionSync` 事件模型，但未明确当前 worker 为 no-op（仅告警后完成）。建议在 `docs/design.md` 标记“未实现/禁用策略”，避免误判系统能力边界。
- SQL `files` 表当前扫描策略（全量递归 + 全量加载）未在文档中披露。建议在架构文档补充“当前复杂度与上限”，便于容量规划。

## What's Good

- `line_count` 计算从直接 `as i32` 改为 `min(i32::MAX)`，修复潜在溢出/截断问题（`write_file`、`append_file` 两处）。
- `FilesTable` 从单条 `get_file` N+1 改为批量读取，方向正确，且在中小规模下能显著降低查询往返次数。
- 本地单测通过：`cargo test -p veda-core -p veda-sql -p veda-store -p veda-server`（未运行 `--ignored` 的外部依赖集成测试）。
