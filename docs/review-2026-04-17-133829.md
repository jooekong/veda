## Verdict

Sprint 1 的 6 项任务功能上都齐了，13 个 mock 测试也都绿。但 `invoke_with_args` 对列输入的处理存在**静默正确性 bug**：当 UDF 以列引用作为参数（`SELECT veda_read(path) FROM files`）时，所有行都会复用 row 0 的结果。现有测试只覆盖了标量常量场景，掩盖了这个问题。在合并前必须修复这个点，且补一个针对列参数场景的测试，否则等于把一个"看起来能用"的能力暴露出去。

## Critical Findings

### 1. UDF 不处理逐行输入（Array 参数），所有行返回 row 0 的结果

- **File:** `crates/veda-sql/src/fs_udf.rs:66-87, 118-177`
- **What:** `extract_string_arg` 无论 `ColumnarValue::Scalar` 还是 `ColumnarValue::Array`，都只取 `a.value(0)`，且 `invoke_with_args` 对任何场景都返回 `ColumnarValue::Scalar`。DataFusion 对返回的标量做广播，等价于"对一个 batch 只执行一次，用 row 0 的参数，对所有行返回同一个结果"。
- **Why it matters:**
  - `SELECT veda_read(path) FROM files` 会返回同一个文件内容填满所有行，用户看不出异常。
  - `SELECT veda_write(path, content) FROM files` 只对 row 0 真正写入一次，其他行被忽略——静默丢数据。
  - `SELECT veda_remove(path) FROM files WHERE xxx` 同理，只删第一条匹配。
  - 这类 bug 在标量常量测试中永远测不到。现有 6 个 UDF 测试全是 `SELECT veda_xxx('/literal')`，所以全绿。
- **Recommendation:**
  1. `invoke_with_args` 里读 `args.number_rows` 和 `args.args[i]` 的数组形态，对每行循环调用 `FsService`，构建对应长度的 `StringArray` / `Int64Array` / `BooleanArray`，返回 `ColumnarValue::Array`。
  2. 对单行批次保留 Scalar 快路径（性能考虑）。
  3. 加测试：`SELECT veda_exists(path) FROM files`（至少 2 行 dentry），断言结果长度与输入一致、值逐行不同。

### 2. 写 UDF 允许在 `SELECT` 中随意副作用，但无保护或审计

- **File:** `crates/veda-sql/src/fs_udf.rs:130-172`
- **What:** `veda_write` / `veda_append` / `veda_remove` / `veda_mkdir` 都是 `Volatility::Volatile`，且在 SELECT 执行路径里直接触发真实的 MySQL 事务、outbox、fs_events。一个误打的 `SELECT veda_remove(path) FROM files WHERE is_dir = true` 在 bug #1 修好之后就能批量删目录。
- **Why it matters:** 现阶段 `/v1/sql` 端点已经对外开放（见 `routes/sql.rs`），SQL 层没有做 DML/DDL 分离、没有 dry-run、没有 audit 日志。一次事故可以静默穿过认证。
- **Recommendation:** 短期至少加两条：
  1. UDF 里 log 每次写操作（路径、workspace_id）；
  2. 在 `AuthWorkspace.read_only` 场景下直接拒绝写 UDF 的 SQL（现在不会拒绝，因为权限校验只在 REST 写路由）。
     中期看 fs9-plan 的 "Phase D 稳定化" 部分，应该补一个"只读 SQL 会话"模式，或把写 UDF 限制在显式 mutation endpoint。

### 3. `veda_remove` 始终返回 1，与方案签名不符

- **File:** `crates/veda-sql/src/fs_udf.rs:163-167`
- **What:** 方案文档（`fs9-plan.md:66`）说 `veda_remove(path)` 语义是"删除文件/目录，返回删除数"。当前实现调用 `FsService::delete`（返回 `()`）之后硬编码返回 1。目录删除真实数量无法从 UDF 感知。
- **Why it matters:** 用户根据返回值做判断（`SELECT veda_remove('/old/')` 期望得到批量删除数）会拿到误导结果。
- **Recommendation:** `FsService::delete` 改为返回 `u64`（受影响 dentry 数），或新增 `delete_count` 方法；UDF 透传返回值。

## Strategic Concerns

### A. UDF 的错误语义一刀切

- **File:** `crates/veda-sql/src/fs_udf.rs:125-162`
- **What:** `veda_read` / `veda_size` / `veda_mtime` 把所有 `FsService` 错误（NotFound、InvalidPath、Storage、PermissionDenied）都统一映射为 NULL；`veda_exists` 把所有错误都当作"不存在"。
- **Why it matters:** 真实的 DB 故障、路径格式错误、workspace 错位等都被伪装成"文件不存在"，线上排查会极困难。
- **Recommendation:** 只把 `VedaError::NotFound` 映射为 NULL / false；其他错误经 `exec_err` 上抛为 `DataFusionError::Execution`，让 SQL 查询失败。这样也和写 UDF 的错误处理一致。

### B. `veda_read` 无大小保护

- **File:** `crates/veda-sql/src/fs_udf.rs:123-128`
- **What:** 读取整个文件进内存，没有 size 预检查，没有配额。单个超大文件（或 `SELECT veda_read(path) FROM files`，遍历 workspace 所有文件）就能打爆 server 内存。
- **Why it matters:** fs9-plan.md:347 已经为 Phase C 规划了 `MAX_SINGLE_FILE_READ = 50MB`，但 Phase 1 的 scalar UDF 同样需要这个保护——而且更紧迫，因为 scalar UDF 已经对外开放。
- **Recommendation:** 在 UDF 调用前先 `stat`，对 `size_bytes > 50MB` 直接返回 `DataFusionError::Execution("file too large")`。或者干脆在 Phase 1 就抽出一个共享常量 `MAX_SINGLE_FILE_READ`。

### C. 每次 `execute` 新建 `SessionContext`，UDF 每次重建

- **File:** `crates/veda-sql/src/engine.rs:37-66`
- **What:** 每个 SQL 请求都走一遍 table 注册 + 8 个 UDF 注册。注册成本小，但在高 QPS 下每次都 `list_collection_schemas` 也是个不必要的 MySQL 往返。
- **Why it matters:** 现阶段影响不大，但 workspace × collection 数量上来后会成为瓶颈。
- **Recommendation:** 可以缓存 `SessionContext` 模板，只在 schema 变更时失效；或退一步，每个 workspace 缓存一个 `SessionContext`。不是 P0，但建议放到 TODO。

### D. 写 UDF 的事务粒度仅到单次调用

- **File:** `crates/veda-core/src/service/fs.rs:23-190, 557-585`
- **What:** 方案文档明确承认这是设计限制（`fs9-plan.md:141-142`），但用户看到 SQL 语法会有"一条语句是原子"的直觉。当一条 SQL 里有多个 `veda_write` 调用，中间失败不会回滚前面的写。
- **Why it matters:** 这是 fs9 架构层面的取舍，短期无法解决。但需要在对外文档里明确。
- **Recommendation:** 在 README 或 SQL 端点文档里加一段"UDF 事务边界"说明。不改代码。

### Key Decisions Observed

- **选择在 SessionContext 级注入 `FsUdfContext`，每次 execute 重建**：与 workspace_id 强绑定，不跨 workspace 污染；代价是 per-query 注册开销。
- **UDF 全部标记 `Volatility::Volatile`**：阻止 DataFusion 把 UDF 调用提升/缓存/优化掉。对写 UDF 必须，对读 UDF 是保守选择（牺牲潜在优化）。正确。
- **写 UDF 在 SELECT 中允许任意副作用**：与 fs9 同构，接受违反 SQL 纯函数语义的代价。
- **`append_file` 采用 read-modify-write 而非原生 append**：与 MySQL inline/chunked 存储模型一致，接受非原子性。方案文档已说明。
- **`collection_table` 数字列类型本地化处理，而非通用 ScalarValue 转换**：为三种主要类型（Int64/Float64/Boolean）写了三个独立循环。未来加类型（Date/Timestamp）需要再加一个分支。可接受。
