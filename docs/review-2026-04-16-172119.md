## Verdict

当前实现**不建议直接上线**。核心链路（FS COW、Outbox 任务恢复、向量同步）存在会导致数据错误或一致性漂移的缺陷，且测试集当前无法完整编译。修复 Critical Findings 后再进入回归测试阶段。

## Critical Findings

### 1) COW 语义被破坏：修改一个副本会污染所有副本
- **File:** `crates/veda-core/src/service/fs.rs:43`
- **What:** `copy_file()` 通过共享 `file_id` + `ref_count` 实现 COW，但 `write_file()` 对已有 `file_id`直接原地覆盖内容与 revision，没有在 `ref_count > 1` 时分叉新文件。
- **Why it matters:** 用户修改 `/a.txt` 会无意修改 `/b.txt`（如果 `/b.txt` 是 copy 得到的共享引用），这是直接的数据正确性问题。
- **Recommendation:** 在 `write_file()` 中，当目标 dentry 对应 file 的 `ref_count > 1` 时：新建 file 记录并写入新内容、将当前 dentry 指向新 file、旧 file `ref_count - 1`；仅 `ref_count == 1` 时允许原地更新。

### 2) ref_count 泄漏：copy 到已指向同一 file 的目标路径会重复加计数
- **File:** `crates/veda-core/src/service/fs.rs:421`
- **What:** `copy_file()` 先 `increment_ref_count(file_id)`，之后若发现 `dst` 已经指向同一 `file_id`，不会回滚这次增加。
- **Why it matters:** 长期会导致 `ref_count` 虚高，删除最后一个真实引用后文件仍无法被清理（存储泄漏、垃圾数据残留）。
- **Recommendation:** 在增计数前先读取/比较 `dst.file_id`；若 `dst` 已等于 `src.file_id` 直接 no-op 返回，或在该分支补偿一次 `decrement_ref_count`。

### 3) 空文件同步会留下旧向量（陈旧搜索结果）
- **File:** `crates/veda-server/src/worker.rs:117`
- **What:** `handle_chunk_sync()` 在 `semantic_chunk` 结果为空时直接 `return Ok(())`，而删除旧向量发生在后面的 `delete_chunks()`。
- **Why it matters:** 文件被清空后，Milvus 仍保留历史向量，搜索会命中已不存在内容，造成“幽灵结果”。
- **Recommendation:** 先执行 `delete_chunks(workspace_id, file_id)`，再根据新内容决定是否 upsert；或在空内容分支显式删向量。

### 4) Outbox 任务不可恢复：租约过期任务不会被重新 claim
- **File:** `crates/veda-store/src/mysql.rs:1018`
- **What:** `claim()` 仅查询 `status = 'pending'`，不会回收 `status='processing' AND lease_until <= now` 的任务。
- **Why it matters:** Worker 异常退出后任务可能永久卡死，向量同步与删除长期不一致。
- **Recommendation:** claim 条件改为“pending 可用 + processing 租约过期”两类并统一抢占；必要时记录 claim owner / attempt。

### 5) 目录递归操作使用未转义 LIKE，`%`/`_` 文件名可误删或误匹配
- **File:** `crates/veda-store/src/mysql.rs:739`
- **What:** `list_dentries_under()` / `delete_dentries_under()` 通过 `path LIKE '{prefix}/%'` 查子树，但没有转义 `%`、`_`。同时路径校验允许这两个字符。
- **Why it matters:** 合法路径名会被当作 SQL 通配符，可能删除或读取到错误目录范围（高风险数据破坏）。
- **Recommendation:** 对 LIKE 参数做转义并使用 `ESCAPE`；或在路径规则中禁止 `%`、`_`。

### 6) 路径唯一约束是前缀索引，不是真唯一
- **File:** `crates/veda-store/src/mysql.rs:342`
- **What:** `UNIQUE INDEX idx_ws_path (workspace_id, path(255))` 只保证 path 前 255 字符唯一。
- **Why it matters:** 长路径在前 255 相同会冲突，出现“不同文件被判重复”或不可预测行为。
- **Recommendation:** 改为可验证的全量唯一策略（例如 path hash 列 + 唯一索引，或调整字符集/索引方案以支持完整 path）。

### 7) SQL collection 查询被静默截断且类型退化为字符串
- **File:** `crates/veda-sql/src/collection_table.rs:26`
- **What:** schema 构建把字段统一映射为 `Utf8`，且 scan 固定 `query_collection(..., 16384)`，忽略 SQL 本身的 filter/limit pushdown。
- **Why it matters:** 数值排序/比较变成字典序，查询语义错误；超过 16384 行会静默丢数据。
- **Recommendation:** 保留真实 Arrow 类型，支持 predicate/limit 下推或分批扫描，禁止硬编码截断。

### 8) 全量测试当前不可编译，CI 基线已失效
- **File:** `crates/veda-store/tests/milvus_test.rs:122`
- **What:** 测试里 `FieldDefinition` 仍使用已移除字段 `embed`，`cargo test --workspace --no-run` 直接失败。
- **Why it matters:** 无法建立可靠回归防线，后续改动缺乏编译级测试保护。
- **Recommendation:** 同步测试数据结构，恢复 `cargo test --workspace --no-run` 绿灯，并把它设为必过门禁。

## Strategic Concerns

### 1) 搜索路径补全吞错，故障会被静默降级
- **File:** `crates/veda-core/src/service/search.rs:79`
- **What:** `get_dentry_path_by_file_id(...).await.unwrap_or(None)` 直接吞掉存储错误。
- **Why it matters:** 后端异常时请求仍返回 200 但结果不完整，且 `path_prefix` 过滤会进一步误删结果，排障困难。
- **Recommendation:** 路径补全失败应返回错误或至少打结构化告警，不应无声吞错。

### 2) `files` 虚表每次全树扫描 + N 次文件查询，扩展性差
- **File:** `crates/veda-sql/src/files_table.rs:69`
- **What:** scan 时递归拉全量目录，再逐文件 `get_file`。
- **Why it matters:** 数据规模上来后 SQL 查询会线性放大并占内存，延迟不可控。
- **Recommendation:** 增加批量加载接口与过滤下推，减少每次 scan 的全量 materialization。

### 3) 设计文档声明 checksum dedup，但实现只做“同路径同内容短路”
- **File:** `crates/veda-core/src/service/fs.rs:85`
- **What:** 新文件写入流程未使用 `find_file_by_checksum()` 做跨路径复用。
- **Why it matters:** 与“content-addressed dedup”目标不一致，存储成本与写放大偏高。
- **Recommendation:** 按 workspace + checksum 做内容复用（配合 ref_count），或下调文档承诺。

### 4) 配置能力与计划文档不一致（缺少 env override）
- **File:** `crates/veda-server/src/config.rs:69`
- **What:** `ServerConfig::load` 只读 TOML，未实现 env override。
- **Why it matters:** 与实施计划描述不一致，部署时可配置性低于预期。
- **Recommendation:** 若需要该能力则补实现；若不需要应更新计划文档状态。
